## 一、为什么必须补这 3 条线

对照赛题原文和评分标准，这三条线关联的分数：

| 遗漏维度 | 关联评分项 | 关联分值 |

## 二、情感线：文本对话情感分析 + 情感趋势报告

### 2.1 开源方案选型

| 方案 | 仓库 | 协议 | 特点 | 适用场景 |
|------|------|------|------|---------|
| **SnowNLP** ⭐推荐 | [isnowfy/snownlp](https://github.com/isnowfy/snownlp) | MIT | `pip install snownlp`，纯 Python，中文原生支持，`s.sentiments` 返回 0~1 正面概率 | 对话文本情感打分 |
| Transformers pipeline | HuggingFace | Apache 2.0 | `pipeline("sentiment-analysis", model="uer/roberta-base-finetuned-jd-binary-chinese")` | 更高精度，需 GPU |
| DeepFace | [serengil/deepface](https://github.com/serengil/deepface) | MIT | 面部表情识别（7种基本表情），适合视频场景 | 游客面部表情分析（可选扩展） |

**推荐首版方案**：SnowNLP。理由：
- `pip install snownlp` 零配置，无模型下载，即装即用
- 中文原生，不需要翻译成英文
- MIT 协议，无版权风险
- 延迟 < 10ms/条，不影响问答主链路
- 对景区场景的正面/负面二分类够用（游客说"太震撼了"→ 0.98，"等了两个小时"→ 0.12）

### 2.2 落地设计

#### 2.2.1 问答对话情感实时标记

在 `QAPipeline` 的问答日志记录环节，对游客 query 和系统 answer 各跑一次 SnowNLP 打情感分：

```python
# backend/app/services/sentiment/analyzer.py
from snownlp import SnowNLP

class SentimentAnalyzer:
    def analyze(self, text: str) -> SentimentResult:
        s = SnowNLP(text)
        score = s.sentiments  # 0~1, >0.5 正面, <0.5 负面
        if score > 0.7:
            label = "positive"
        elif score < 0.3:
            label = "negative"
        else:
            label = "neutral"
        return SentimentResult(score=score, label=label)
```

修改 `pipeline.py` 的日志记录处（第 126-135 行和第 158-167 行），在 `record_to_log` 前加情感标记：

```python
sentiment = sentiment_analyzer.analyze(request.query)
self.chat_log_repository.create(
    ...,
    sentiment_score=sentiment.score,
    sentiment_label=sentiment.label,
)
```

#### 2.2.2 模型字段扩展

在 `models/chat_log.py` 的 `ChatLog` 表加两个字段：

```python
sentiment_score: Mapped[float | None] = mapped_column(Float, nullable=True)
sentiment_label: Mapped[str | None] = mapped_column(String(20), nullable=True)
```

#### 2.2.3 后台情感趋势报告

新增 API：`GET /api/admin/sentiment-trend`

```python
@router.get("/sentiment-trend", response_model=SentimentTrendResponse)
async def sentiment_trend(
    days: int = 7,
    session: Session = Depends(get_db_session),
) -> SentimentTrendResponse:
    logs = ChatLogRepository(session).list_recent(days=days)
    daily_stats = defaultdict(lambda: {"positive": 0, "neutral": 0, "negative": 0, "avg_score": 0.0})
    for log in logs:
        day = log.created_at.strftime("%Y-%m-%d")
        label = log.sentiment_label or "neutral"
        daily_stats[day][label] += 1
    # ... aggregate and return
```

前端管理后台新增一个"游客感受度报告"卡片，展示：
- 近 7 天情感趋势曲线（正面/中性/负面占比）
- 今日平均情感分
- 情感最低的 Top 5 对话（供管理方定位问题）

#### 2.2.4 数字人表情联动（文档层面，本轮不实现）

在架构文档中声明：情感分析结果可同步驱动数字人表情：
- `sentiment_score > 0.7` → 数字人微笑表情
- `sentiment_score < 0.3` → 数字人关切表情
- 对话内容含"感谢""太棒了"等 → 数字人开心 + 点头

> 本轮只做情感标记和后台报告。表情联动标记在架构文档中作为"多模态情感分析"的体现，供演示视频展示设计思路。

## 四、GPS 加分项：弱信号场景下的位置感知方案

### 4.1 赛题原文

> "(可选)鼓励团队考虑更复杂的现实场景，如考虑GPS信号或者难以定位的情况，可提出解决方案。"

### 4.2 方案设计（仅文档层面，零代码实现）

#### 方案名称：景区级混合定位与手动触发双模方案

#### 适用场景

灵山胜境多处景点（梵宫内部、五印坛城内部、祥符禅寺屋檐下）存在 GPS 信号弱或完全丢失的情况。游客在这些区域无法自动触发基于位置的导览讲解。

#### 方案架构

```
GPS 可用时：
  手机 GPS → 判断游客所在景点区域 → 自动触发该景点讲解
                                    → 推送"您已到达XX景点"

GPS 不可用时（三级 fallback）：
  L1: WiFi 指纹定位 → 检测景区 AP 信号强度特征匹配
  L2: 蓝牙信标       → 各景点部署低功耗 BLE Beacon
  L3: 手动选择       → 游客在 App 中选择"我在哪个景点附近"
```

#### 各方案对比

| 方案 | 精度 | 成本 | 部署难度 | 适用景点 |
|------|------|------|---------|---------|
| GPS | 5-10m | 零 | 零 | 室外开阔区域（灵山大佛广场、菩提大道） |
| WiFi 指纹 | 3-5m | 低 | 中（需提前采集指纹地图） | 室内/半室内（梵宫、祥符禅寺） |
| 蓝牙 Beacon | 1-3m | 中（硬件采购） | 低 | 室内精确点位（梵宫各展厅） |
| 手动选择 | 依赖游客 | 零 | 零 | 所有场景兜底 |

#### 推荐落地策略（比赛阶段）

1. **GPS + 手动选择双模**（已在当前架构中实现基础）
2. **WiFi 指纹定位**：在方案 PPT 中描述原理（RSSI 信号强度 + KNN 匹配），作为创新技术亮点
3. **蓝牙 Beacon**：作为可扩展方案标记
4. 在系统中预留 `LocationProvider` 抽象接口，现有实现仅 GPS，接口设计兼容未来 WiFi/蓝牙方案

```python
# 接口预留（写入设计文档，不实现）
class LocationProvider(ABC):
    @abstractmethod
    def get_current_location(self) -> Optional[Location]: ...

class GPSLocationProvider(LocationProvider): ...     # 已实现
class WiFiFingerprintProvider(LocationProvider): ... # 预留
class BeaconLocationProvider(LocationProvider): ...  # 预留

```
