## 二、补丁 1：FAQ 快速命中层

### 2.1 为什么必须加

景区场景 60% 以上的问题是高频重复的（"大佛多高""门票多少"）。走完整 RAG + LLM 链路延迟高且不稳定，FAQ 命中后 < 50ms 返回，同时答案 100% 可控。

### 2.2 文件位置

```
backend/app/services/qa/
├── __init__.py
├── pipeline.py          # 问答编排主流程（含FAQ→RAG→LLM fallback）
├── faq_matcher.py       # 🟢 FAQ 匹配器
└── cache.py             # 🟢 问答缓存（补丁4）
```

### 2.3 FAQ 匹配策略（三级 fallback）

| 级别 | 策略 | 延迟 | 适用场景 |
|------|------|------|---------|
| L1 精确匹配 | query 归一化后 hash 查表 | < 5ms | "灵山大佛多高" |
| L2 模糊匹配 | BM25 / jieba 分词后布尔检索 | < 20ms | "大佛的高度是多少" |
| L3 语义匹配 | query embedding 向量相似度 | < 100ms | "那个最高的佛像有多高" |

**L1/L2 命中直接返回答案，不走 LLM。**

### 2.4 FAQ 数据来源

从景区资料包（docx）中结构化提取，格式：

```json
{
  "id": "faq_001",
  "category": "景点信息",
  "aliases": ["灵山大佛多高", "大佛高度", "灵山大佛高度多少米"],
  "answer": "灵山大佛通高88米，其中佛体高79米，莲花瓣高9米，总高101.5米（含基座），是世界上最高的露天青铜释迦牟尼立像。",
  "sources": ["灵山胜境：历史、文化、景点特色与个性化游览指南.docx - 灵山大佛章节"]
}
```

### 2.5 接口约定

```python
class FAQMatcher:
    def add(self, faq_entries: list[FAQEntry]) -> None: ...
    def match_exact(self, query: str) -> Optional[FAQEntry]: ...
    def match_fuzzy(self, query: str) -> Optional[FAQEntry]: ...
    def match_semantic(self, query: str, embedding_fn) -> Optional[FAQEntry]: ...
    def match(self, query: str, embedding_fn) -> FAQMatchResult:
        """三级 fallback，返回匹配结果 + 命中级别"""
```

### 3.2 Query 改写

**场景**：游客说"那个大佛是啥时候建的来着" → 改写为"灵山大佛建造时间"

策略（不依赖 LLM，保持低延迟）：

| 改写步骤 | 方法 |
|----------|------|
| 实体替换 | 维护别名映射表，"大佛" → "灵山大佛"，"那个塔" → "五印坛城" |
| 口语化转书面 | 正则替换，"啥时候"→"时间"，"来着"→删除 |
| 补全主语 | 缺主语时从对话上下文补全 |

```python
class QueryRewriter:
    def rewrite(self, raw_query: str, context: Optional[ConversationContext] = None) -> str:
        """返回规范化 query"""
```

### 3.3 Reranker

**为什么 ChromaDB 不够**：ChromaDB 用 embedding 向量相似度检索，中文场景 Recall@5 通常只有 70-80%。加一层 Cross-Encoder Reranker 可提升到 95%+。

**首版方案**：`BAAI/bge-reranker-base`（~270MB，CPU 推理 < 100ms）

```python
class Reranker:
    def rerank(self, query: str, documents: list[str], top_k: int = 5) -> list[RerankResult]:
        """
        输入：query + ChromaDB 返回的 Top-20 文档
        输出：重排后的 Top-5，含相关性分数
        """
```

### 3.4 RAG 检索完整流程

```python
async def retrieve(self, query: str, top_k: int = 5) -> list[Document]:
    # Step 1: Query 改写
    normalized = self.query_rewriter.rewrite(query)
    
    # Step 2: ChromaDB 向量检索 Top-20
    candidates = self.chroma_db.query(normalized, n_results=20)
    
    # Step 3: Reranker 重排 → Top-5
    ranked = self.reranker.rerank(query, candidates, top_k=top_k)
    
    return ranked
```

## 四、补丁 3：全链路流式编排

### 4.1 为什么必须流式

| 阶段 | 非流式延迟 | 流式延迟 |
|------|-----------|---------|
| LLM 生成 | 等待全部 token 生成完（~4-6s） | 首 token ~1s |
| TTS 合成 | 等 LLM 全部结束后合成（+2s） | LLM 出首句即合成（~500ms） |
| **总计** | **6-8 秒** ❌ | **~3 秒** ✅ |

### 4.2 流式编排主流程

```python
# backend/app/services/qa/pipeline.py

async def streaming_qa_pipeline(
    query: str,
    asr_result: ASRResult,
    conversation_context: ConversationContext,
) -> AsyncGenerator[StreamEvent, None]:
    """
    流式问答编排。
    产出 StreamEvent 序列：文本块、音频块、状态变化、检索来源。
    """
    
    # Step 0: Query 改写
    normalized_query = query_rewriter.rewrite(query, conversation_context)
    
    # Step 1: FAQ 快速命中（同步，极快）
    yield StreamEvent(type="status", data="正在查找答案...")
    faq_result = faq_matcher.match(normalized_query, embed)
    if faq_result.is_hit:
        yield StreamEvent(type="text", data=faq_result.answer)
        yield StreamEvent(type="sources", data=faq_result.sources)
        audio = await tts.synthesize(faq_result.answer)  # 整段合成
        yield StreamEvent(type="audio", data=audio)
        record_to_log(query, faq_result.answer, faq_result.sources, hit_level=faq_result.level)
        return
    
    # Step 2: RAG 检索 + Reranker
    yield StreamEvent(type="status", data="正在检索景区知识...")
    docs = await rag.retrieve(normalized_query)
    yield StreamEvent(type="sources", data=format_sources(docs))
    
    # Step 3: LLM 流式生成 + TTS 句级并行
    sentence_buffer = ""
    async for token in llm_service.stream_generate(query, docs):
        sentence_buffer += token
        yield StreamEvent(type="text_chunk", data=token)  # 前端逐字展示
        
        if is_sentence_boundary(token):  # 遇到句号、问号等
            sentence = sentence_buffer.strip()
            sentence_buffer = ""
            # TTS 异步合成当前句子（不阻塞 LLM 继续生成）
            asyncio.create_task(synthesize_and_yield(sentence, event_queue))
    
    # 处理残留内容
    if sentence_buffer.strip():
        asyncio.create_task(synthesize_and_yield(sentence_buffer, event_queue))
    
    # 等待所有 TTS 完成
    await drain_audio_queue(event_queue)
    
    # 记录到问答日志（供评测和后台分析）
    record_to_log(query, full_answer, docs)
```

### 4.3 SSE 传输格式

前端通过 `EventSource` 或 `fetch` + `ReadableStream` 接收：

```
event: status
data: {"text": "正在查找答案..."}

event: sources
data: {"docs": [{"title": "灵山大佛", "snippet": "...", "source": "..."}]}

event: text_chunk
data: {"token": "灵", "is_first": true}

event: text_chunk
data: {"token": "山"}

event: audio
data: {"base64": "...", "duration_ms": 2500}

event: text
data: {"text": "灵山大佛通高88米...", "is_complete": true}

event: done
data: {}
```

### 4.4 降级策略

| 场景 | 处理 |
|------|------|
| ASR 不可用 | 退化为纯文本输入 |
| TTS 不可用 | 仅返回文本回答，前端不播放语音 |
| LLM 不可用 | FAQ 命中可独立工作；FAQ 未命中返回"服务暂不可用" |
| Avatar 不可用 | 前端隐藏数字人区域，保留文本+语音交互 |

## 五、补丁 4：问答缓存

### 5.1 缓存层级

| 层级 | 范围 | 存储 | TTL | 命中延迟 |
|------|------|------|-----|---------|
| L1 内存缓存 | 进程内 LRU | Python dict | 会话期间 | < 1ms |
| L2 持久化缓存 | 归一化 query → answer | SQLite | 7天 | < 10ms |

## 九、关键约束再次强调

| 约束 | 含义 |
|------|------|
| **每个 stub/真实实现共用同一个 base 接口** | 后续切换不伤筋骨 |
| **问答全链路必须基于 SSE** | 不用 WebSocket 做大炮打蚊子 |
| **LLM 出完整句子后立即送 TTS** | 不等整段生成完 |
| **所有外部服务做成 async** | ASR/TTS/LLM 都走 asyncio |
| **compose 路径至少读完 base.py 再写实现** | 保证接口一致性 |
