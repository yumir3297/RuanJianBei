## 二、最终目标与不可退让的指标

最终作品不是单一问答页面，而是一套面向真实景区的可运营 AI 数字人导览系统，包含游客交互端和管理后台。

必须满足：

1. 至少接入 1 个真正参与业务的多模态大模型。
2. 景区事实性问答准确率不低于 90%，并由固定评测集证明。
3. 语音问答端到端响应目标小于 5 秒。
4. 数字人具备语音播报、口型同步和状态/情感表情联动。
5. 核心景区事实以官方资料包为第一事实源。
6. 游客端支持文本、语音、图片识别和主动选择等交互入口。
7. 管理后台支持知识管理、数字人配置、游客分析和数据大屏。
8. 全系统具备证据来源、降级通道和可回归验证。

## 三、设计原则

### 3.1 保留的顶层原则

1. `Retrieval-First`：事实回答先检索，再生成。
2. `Full-Streaming`：文本、TTS 和数字人状态尽量流式衔接。
3. `Graceful Degradation`：图片、语音、模型或 Avatar 失败时，文本问答仍可使用。
4. `Official-Data-First`：选择按钮、模型常识和历史对话都不能成为景区事实来源。
5. `Smallest Correct Change`：不为尚不存在的规模提前建设复杂平台。
6. `Measurable Before Claim`：命中率、准确率和延迟只认实测结果。

### 3.2 对 09 文档的最终取舍

采纳：

1. 使用结构化 `SelectionContext`，不使用两个平铺字符串字段。
2. 使用明确的上下文冲突优先级。
3. 增加受控快捷追问与选择面包屑。
4. 选择上下文进入缓存键，但不作为事实证据。
5. 不引入 Redis、乐观锁和五层编排抽象。
6. 主动选择与 Reranker 性能优化并行推进。
7. 补齐 FAQ L3 与知识盲区闭环。

修正：

1. 不把全部业务逻辑堆进 `pipeline.py`。`QAPipeline` 只负责编排，选择解析、检索、LLM、ASR、TTS、Avatar 继续保持现有模块边界。
2. 不新增重复保存景点事实的 `QuickSpot` 数据。景点选项来自 `KnowledgeEntry`，路线选项来自 `RouteTemplate`。
3. 仅新增 `QuickTopic` 配置数据，避免把话题按钮写死在前端。
4. 选择通过结构化 metadata/filter 限定检索，不把 `search_hint` 粗暴拼入游客问题。
5. 前端携带上下文，但服务端每轮都要校验实体 ID、解析冲突并返回新的可信上下文。
6. 方言改写不能在置信度不足时替游客猜答案，只能生成候选并请求确认。
7. AnswerValidator 不能被描述成绝对防幻觉机制；核心保障是检索证据、无证据拒答、受约束 Prompt、引用与评测。

拒绝：

1. 未经数据证明的“FAQ 命中 60%/75%”和推测延迟。
2. 仅使用多模态模型的文本接口就宣称满足多模态要求。
3. 在 CPU 上对每个 RAG 请求固定执行 Top-20 CrossEncoder。
4. 一次性修改二十多个文件、同时引入多套外部系统。
5. 依靠 `create_all` 给已有表自动增加字段。

## 四、最终系统架构

```text
文本输入 ───────────────────────────────┐
语音输入 -> ASR -> 置信度/候选确认 ──────┤
图片输入 -> 多模态模型识别景点候选 ──────┤
主动选择 -> 景点/话题/路线结构化上下文 ───┤
                                         v
                         交互上下文解析与冲突消解
                                         |
                              QueryRewriter 规范化
                                         |
                   FAQ L1/L2/L3 -> 选择感知缓存
                              | 未命中
                              v
             Chroma metadata 检索 -> 条件式 Reranker
                              |
                       官方证据与来源编号
                              |
                   多模态 LLM 受约束流式生成
                              |
           SSE 文本/来源/上下文/快捷追问/状态事件
                              |
                 句级 TTS -> Avatar 口型与表情
                              |
                    ChatLog / 盲区 / 运营统计
```

### 4.1 架构边界

`QAPipeline`：

1. 串联现有服务。
2. 决定 FAQ、缓存、RAG、推荐等当前三类路径。
3. 发送 SSE 事件并记录日志。
4. 不直接实现向量检索、模型 SDK、语音算法或 Avatar 渲染。

独立可替换服务：

1. `services/rag/`：Embedding、Chroma、条件式重排。
2. `services/llm/`：OpenAI-compatible 多模态模型适配。
3. `services/asr/`：语音识别、热词和置信度输出。
4. `services/tts/`：句级语音合成及时间信息。
5. `services/avatar/`：状态、口型和表情控制。
6. `services/recommend/`：结构化路线推荐。
7. `services/qa/`：FAQ、缓存、主动选择解析和主链路编排。

本项目不创建通用工作流引擎，也不创建 Orchestrator/DomainRouter/EvidenceGate/AnswerPolicy 五层文件体系。

## 五、游客主动选择与上下文设计

### 5.1 SelectionContext

建议最终请求结构：

```python
class SelectionContext(BaseModel):
    mode: Literal["free_chat", "topic", "attraction", "route"] = "free_chat"
    attraction_id: int | None = None
    topic_key: str | None = None
    route_id: str | None = None
    interests: list[str] = Field(default_factory=list)
    audience_type: str | None = None
    available_hours: int | None = None
    avoid_crowded: bool | None = None
```

设计约束：

1. 不传景点标题作为可信字段，由服务端根据 `attraction_id` 查询。
2. 景点列表来自 `knowledge_entries`，路线来自 `route_templates`。
3. `quick_topics` 只保存话题 UI 配置和检索分类，不保存景区事实。
4. 前端通过 `localStorage` 恢复选择；服务端不在 MVP 阶段保存 Session 状态。
5. 服务端返回 `context` SSE 事件，前端下一轮原样携带；服务端仍需重新校验。

### 5.2 冲突优先级

1. 本轮文字中明确出现且能在知识库验证的实体。
2. 本轮主动点击的选择。
3. 上一轮服务端返回的 active entity。
4. 前端历史摘要。
5. 系统默认范围。

任何来源都只能限定检索范围，不能替代官方证据。

### 5.3 主动选择交互

游客端提供：

1. 入口目标：了解景点、规划路线、实用信息。
2. 景点选择：读取真实知识条目。
3. 话题选择：历史文化、建筑艺术、祈福体验、亲子等后台配置项。
4. 路线选择：读取真实路线模板。
5. 面包屑：展示当前景点、话题或路线。
6. 快捷追问：只由受控模板和知识元数据生成 2-4 项。

## 六、准确性与鲁棒性方案

### 6.1 两级快速通道

1. FAQ L1 精确匹配。
2. FAQ L2 模糊匹配。
3. FAQ L3 预计算向量语义匹配。
4. 选择感知问答缓存。
5. 未命中才进入 RAG 与 LLM。

FAQ L3 必须离线或启动时预计算 FAQ 向量矩阵，阈值由评测集校准，不在请求内逐条编码。

### 6.2 RAG 与条件式 Reranker

当前实测：

1. Chroma 原始 Recall@1 为 96.67%。
2. Chroma 原始 Recall@5 为 100%。
3. CrossEncoder 后 Recall@1 为 100%。
4. CrossEncoder CPU P50 为 4731.09ms，P95 为 6079.41ms。

因此最终策略不是删除 Reranker，也不是每次强制执行，而是条件式执行：

1. FAQ、缓存和推荐路径不执行 Reranker。
2. 主动选择已将候选限定到单一景点且向量结果置信度足够时跳过。
3. 仅对歧义问题、低分差问题和跨景点问题执行真实 Reranker。
4. 优先评测 Top-5/Top-8、较短 `max_length` 和批量参数。
5. 超过阶段预算立即降级为 BM25 或原向量顺序。
6. 阈值和参数只能由 30/100 题评测确定。

阶段目标：保住 Recall@5 100%，同时使进入 CrossEncoder 的请求比例和耗时不再破坏 5 秒语音总预算。

### 6.3 生成约束与证据链

1. 没有检索证据时不调用 LLM，直接说明资料不足。
2. Prompt 明确限制只能使用编号证据。
3. 回答携带可点击或可展开的来源。
4. 时间、数字、地点等高风险内容优先做结构化一致性检查。
5. 事后验证用于审计、评测和必要的整段拦截，不把流式输出后的检查包装成安全保证。
6. 选择按钮、图像识别结果和 ASR 文本都只是检索线索，不是事实来源。

### 6.4 知识盲区闭环

新增最小 `KnowledgeBlindSpot` 记录：问题、规范化问题、频次、状态、首次/最近时间和解决关联。

闭环流程：

1. 无证据、低置信度或用户负反馈时记录/累加。
2. 后台按频次和最近时间查看。
3. 管理员补充答案后写入 FAQ/知识条目。
4. 自动刷新 FAQ 内存索引；涉及长知识时显式触发 Chroma 重建。
5. 已解决盲区保留审计记录，不直接删除。

## 七、真正的多模态能力

赛题的“至少一个多模态大模型”必须在演示业务中真实使用。

首版落地方式：

1. 使用现有 OpenAI-compatible 适配层接入可处理图像的国产多模态模型。
2. 游客可上传或拍摄景点图片。
3. 模型只输出景点候选、可见特征和置信度，不直接编造景区讲解。
4. 候选映射到 `KnowledgeEntry` 后转为 `SelectionContext`。
5. 最终讲解仍走 FAQ/RAG/官方证据链。
6. 低置信度时展示多个候选让游客确认。

首选模型可为 `Qwen-VL-Plus`，备选本地模型需另做资源评估。实际供应商、账号和费用在代码接入前由用户确认。

## 八、语音与方言方案

### 8.1 ASR

1. 前端真实录音并上传。
2. ASR Provider 返回文本、置信度、候选和耗时。
3. 使用“灵山、大佛、九龙灌浴、梵宫、五印坛城”等景区热词。
4. 文本输入和主动选景点始终作为降级通道。

置信度策略：

1. 高置信度且景点实体可验证：直接进入问答。
2. 中置信度：结合热词和 N-best 生成候选，前端确认后再问答。
3. 低置信度：不自动猜测，要求重说、打字或手动选择。

LLM 方言改写只能用于候选规范化，不能在景点实体不确定时直接替用户决定问题。

### 8.2 TTS 与数字人

1. LLM 逐句输出后启动句级 TTS。
2. TTS Provider 优先返回音频时长及可用的 phoneme/viseme 时间信息。
3. Avatar 优先使用真实时间信息驱动口型；不可用时使用 Web Audio 音量分析降级。
4. Avatar 状态至少包含 `idle`、`listening`、`thinking`、`speaking`、`happy`、`apology`。
5. Avatar 故障不得阻塞文本与音频回答。

首版先完成稳定的轻量 2D Avatar，再决定是否接入 Duix-Avatar 等重型方案。

## 九、管理后台最终范围

必须形成可演示闭环：

1. 知识库：列表、新增、编辑、删除/禁用、资料同步、索引重建。
2. FAQ：维护、启停、别名、L3 索引刷新。
3. 盲区管理：高频未知问题、补充答案、处理状态。
4. 数字人配置：形象、服装主题、声音、表情策略、Provider 状态。
5. 数据大屏：服务次数、热门问题、FAQ/RAG/缓存占比、平均/P95 延迟。
6. 游客感受度：关注点、文本/语音情感趋势、负反馈和服务建议。

情感分析结果只用于语气、表情和运营统计，不参与景区事实判断。具体情感模型须先做小样本验证，暂不直接锁定 SnowNLP。

## 十、缓存、并发与降级

### 10.1 缓存键

缓存键至少包含：

```text
normalized_query + route_domain + attraction_id + topic_key + route_id + knowledge_version
```

知识更新后通过 `knowledge_version` 或显式失效避免返回旧答案。

### 10.2 并发与超时

1. FAQ、Chroma、Reranker、LLM、ASR、TTS 分别记录耗时。
2. Reranker 和 LLM 设置独立并发限制，不能只限制 LLM。
3. 外部 Provider 必须有连接、首包和总超时。
4. 过载时优先返回 FAQ、缓存、原向量证据摘要或文字回答。
5. Redis/SingleFlight 暂不进入 MVP，只有单机压测证明需要后再引入。

## 十一、最终开发主线

### 已完成基线

1. 前后端骨架与 SSE 文本链路。
2. 官方资料提取、清洗、入库和同步。
3. FAQ L1/L2、数据库驱动路线推荐、后台概览。
4. KnowledgeChunk、BGE Embedding、Chroma 真实索引与检索。
5. CrossEncoder、BM25 和原向量顺序三级降级。
6. 30 题检索与重排评测。

### 主线阶段 A：鲁棒交互基础，下一阶段

轨道 A1：主动选择最小闭环

1. `SelectionContext` 与请求扩展。
2. `QuickTopic` 配置表；景点/路线复用现有数据表。
3. `/api/quick-select/bootstrap`。
4. 前端目标、景点、话题、路线选择和面包屑。
5. 服务端冲突解析、metadata 检索过滤、上下文 SSE 事件。
6. 受控快捷追问与选择感知缓存键。

轨道 A2：Reranker 性能闸门

1. 建立跳过/执行条件。
2. 评测 candidate_k、max_length、batch_size。
3. 保留准确率并形成新的 P50/P95 报告。
4. 达不到预算时使用明确降级，不静默拖慢全链路。

轨道 A3：FAQ L3 与盲区闭环

1. FAQ 预计算语义索引和阈值评测。
2. `KnowledgeBlindSpot` 记录与聚合。
3. 管理端查看、补充、刷新 FAQ/索引。

每个轨道内部逐 Step 完成、逐 Step 测试和审查；三个轨道可以交替推进，但不得一次性混改。

### 主线阶段 B：真实多模态 LLM 与证据回答

1. 用户确认模型供应商、API 账号和成本。
2. 替换当前 LLM stub，保留 SSE。
3. 接入证据编号 Prompt、无证据拒答和来源展示。
4. 增加图片上传/拍摄和景点候选识别。
5. 图片候选确认后进入 SelectionContext + RAG。
6. 建立 100 题端到端问答评测和图片识别小样本集。

### 主线阶段 C：管理后台可运营闭环

1. 完成知识 CRUD、同步和重建索引页面。
2. 完成盲区处理页面。
3. 完成数字人配置页面。
4. 完成真实日志驱动的数据大屏与感受度报告。

### 主线阶段 D：语音闭环

1. 真实录音与 ASR endpoint。
2. 热词、置信度和候选确认。
3. 句级流式 TTS。
4. 统计 ASR、检索、LLM、TTS 各段延迟。
5. 端到端 P95 向 5 秒以内收敛。

### 主线阶段 E：数字人闭环

1. 轻量 2D Avatar 状态机。
2. TTS 时间信息或音量驱动口型。
3. 情感/回答状态驱动表情。
4. 后台切换形象、声音和表情策略。
5. 评估是否需要 Duix-Avatar 增强，不破坏已有降级链路。

### 主线阶段 F：最终评测与演示保障

1. 100 题事实/FAQ/推荐评测，准确率不低于 90%。
2. 选择冲突、5 轮追问、知识盲区和无答案测试。
3. 文本、图片、语音和 Avatar 降级测试。
4. FAQ/RAG/LLM/语音全链路 P50/P95。
5. 验证后台数据与游客端真实日志一致。
6. 固化可支撑 7 分钟演示的稳定功能路径。

GPS/WiFi 弱信号定位、复杂 3D Avatar、实时天气/交通/票务和多智能体不进入必做主线，只能在所有硬指标通过后作为加分项评估。

## 十三、执行纪律

1. 每次正式编码前先生成或更新本轮任务总结，写清目标、影响范围、风险和验收。
2. 修改公共 API、数据库 Schema、依赖或模型方案前必须获得用户确认。
3. 所有模型、缓存和新增大型依赖继续放在 D 盘。
4. 每完成一个 Step 就运行测试并做一次需求/架构/赛题要求审查。
5. 遇到走不通的供应商账号、下载、依赖或性能路线时立即停止并向用户说明，不擅自换方案。
6. 本文件批准的是开发顺序和设计边界，不代表已经批准任何具体模型付费、依赖安装或数据库修改。
