# 管理后台特色展示数据说明

## 1. 这批数据要讲什么

这不是一份把所有指标做成“满分”的装饰数据，而是一组用于演示系统真实能力链的确定性模拟数据。它重点展示：

1. FAQ 精确、模糊、语义匹配与缓存承担高频稳定问题；
2. RAG 负责文化讲解、个性化路线和复杂问答，并保留可核验来源；
3. 拍照识景只提供候选景点线索，最终事实仍从景区知识资料中检索；
4. 语音咨询同时记录文字与声音情绪，驱动澄清、安抚和安全优先等响应策略；
5. 对实时班次、临时停演等缺乏可靠证据的问题不猜测，自动沉淀为知识盲区；
6. 管理员可把高频盲区补成 FAQ，形成“发现—补录—重新命中”的运营闭环。

## 2. 与旧演示数据相比的改进

旧脚本 `bootstrap_demo_data.py` 使用未固定的随机数，重复环境下的数据量、命中比例和响应时间会变化；新增情绪字段也基本使用默认值，无法支撑多模态情感分析页面。

新脚本 `bootstrap_showcase_data.py` 采用固定场景与固定分布：

- 近 30 天共 495 条问答，趋势缓慢上升；
- 40 个游客画像，覆盖亲子、文化、休闲、自由行和综合游客；
- 同时覆盖 FAQ、缓存、RAG 和证据不足分支；
- 文本与语音两种输入均有数据，六类情绪均有样本；
- 保留少量模态差异、负反馈和证据不足，数据不会显得失真；
- 5 条待处理盲区与 3 条已解决盲区，能展示知识运营闭环；
- 所有记录使用 `showcase-` 前缀或明确的展示分类，可单独重建，不删除真实数据。

## 3. 使用方法

在 `backend` 目录执行。

先只预览，不写数据库：

```powershell
python -m app.db.bootstrap_showcase_data --dry-run
```

首次写入：

```powershell
python -m app.db.bootstrap_showcase_data
```

已写入后重新生成近 30 天数据：

```powershell
python -m app.db.bootstrap_showcase_data --replace
```

`--replace` 只处理 `showcase-visitor-*` 会话、`showcase-faq-*` FAQ 和脚本中列出的展示盲区，不会清空普通问答、游客画像或正式知识。

建议在启动后端前注入；如果后端已经运行，注入后重启一次，使新增的展示 FAQ 进入运行时匹配器。

## 4. 推荐录制顺序

1. 打开“数据分析”，先展示 30 天咨询趋势、FAQ/RAG 分层命中和真实景点关注度。
2. 展示“游客情绪与服务响应”：指出语音融合量、模态差异和需关注咨询，再展开最近记录中的“澄清引导”“安抚说明”“安全优先”。
3. 展示游客群体与偏好标签，说明路线推荐不是固定模板，而是使用游客类型、兴趣和时间等上下文。
4. 展示高频知识盲区，选择“今日无障碍接驳车实时班次”说明系统为什么拒绝猜测实时信息。
5. 切换到已解决盲区，展示雨天路线、人工讲解和婴儿车咨询已通过 FAQ 完成闭环。
6. 回到游客端，依次演示高频事实、文化讲解、语音焦虑咨询、拍照识景和实时信息拒答。

## 5. 答辩口径与边界

- 应明确称为“评委演示数据”或“模拟运营样本”，不要宣称为真实游客统计。
- 景区开放时间、票价、班次、演出和库存均可能变化；展示答案采用“以当日官方公告或现场口径为准”的保守表达。
- 脚本中的 3 条 `showcase-faq-*` 仅用于演示盲区转 FAQ 的机制，正式部署前应删除或替换为景区审核过的口径。
- 平均响应时间是数据库中全部日志的综合结果。如果环境里还保留旧随机数据，最终页面数值会与脚本的单批预览值不同，这是正常现象。
