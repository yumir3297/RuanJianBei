# 架构决策记录

- 首版骨架采用 `Vue 3 + FastAPI + SQLite + ChromaDB`；ChromaDB 已由预留升级为真实索引与检索。
- 问答主链路固定包含：`Query 改写 -> FAQ -> Cache -> RAG -> LLM -> TTS -> Avatar`。
- 流式接口采用 `SSE`，前端通过 `fetch + ReadableStream` 消费。
- 数据库初始化首轮采用 `create_all`，暂不上迁移体系。
- 比赛提供的 `xlsx` 行为分析数据不作为灵山胜境问答事实源，只作为后台游客行为分析示例数据使用。

## 2026-06-12 最终设计收敛

- 后续开发主线以 `设计文档/10-最终总体设计与开发主线.md` 为准；此前计划保留为历史记录。
- 保持 `QAPipeline` 为轻量编排层，不新增 Orchestrator/DomainRouter/EvidenceGate/AnswerPolicy 五层体系，也不把模型和检索实现塞入 pipeline。
- 游客主动选择采用结构化 `SelectionContext`；景点复用 `KnowledgeEntry`，路线复用 `RouteTemplate`，仅话题使用可配置数据。
- MVP 会话状态采用“前端携带、服务端逐轮验证并返回”的方式，不引入 Redis、服务端 Session 表或 revision 乐观锁。
- 主动选择、图片识别和 ASR 文本只能限定检索，不能充当景区事实证据。
- Reranker 保留但改为条件式执行；FAQ、缓存、高置信度选择范围不固定进入 CrossEncoder。
- 多模态要求通过真实图片识别业务落地：多模态模型识别景点候选，最终回答仍由官方知识 RAG 提供证据。
- FAQ L3 使用预计算向量；命中率和阈值必须以评测为依据。
- 新增知识盲区闭环，管理员补充内容后刷新 FAQ 或重建 RAG 索引。
- 首版数字人优先采用稳定轻量 2D 方案，TTS/Avatar 继续使用可替换 Provider；重型数字人只作为后续增强。
- 新表可继续由 `create_all` 创建；不得依赖 `create_all` 修改已有表字段。

## 2026-06-15 FAQ L3 评测决策

- 拒绝未通过评测的全库纯向量 Top-1 方案，不使用单一阈值解决同景点多意图问题。
- FAQ L3 固定采用“可信实体 + 受控意图双门控 -> 候选内语义排序”。
- `BAAI/bge-small-zh-v1.5` 继续从 D 盘离线缓存加载，与向量 RAG 共享实例。
- 经 21 条正例和 13 条盲区负例评测，运行阈值采用 `0.60`；该阈值只在双门控通过后生效。
- 无可信实体、无受控意图、无对应分类 FAQ 或低于阈值时继续进入 RAG，不以相似 FAQ 补造事实。

## 2026-06-15 知识盲区闭环决策

- 仅在 RAG 无证据时记录盲区，运营记录故障不得阻断游客回答。
- 盲区按规范化问题唯一聚合，每条最多保留 5 个去重原始样本。
- 第一版只实现管理查询和 `resolve-with-faq`；`link-knowledge` 与 Chroma 重建分离。
- 管理员补录 FAQ 必须提供非空来源，系统不自动生成事实答案。
- 已解决盲区保留审计记录，不物理删除，也不在盲区表复制最终答案。
- 官方资料同步必须保护已解决盲区引用的人工 FAQ。
