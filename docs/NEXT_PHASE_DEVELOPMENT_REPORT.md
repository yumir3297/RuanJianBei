## 三、对 DeepSeek 审查建议的可行性分析

### 3.1 建议：真实 Reranker 必须同步落地

**判断：完全可采纳**

原因：

1. 与顶层设计文档完全一致。
2. 当前 `Reranker` 确实是假实现，保留它会导致 Chroma 检索结果没有真正的精排层。
3. 如果目标是逼近 `90%` 准确率，真实重排几乎是必需项。

**计划结论**：

- 下一阶段 Phase 1 不再单独写“RAG”，而是写成：
  - `真实 RAG + 真实 Reranker`

### 3.2 建议：明确 Embedding 模型选型

**判断：完全可采纳**

原因：

1. ChromaDB 不可能脱离 embedding 模型存在。
2. 顶层设计文档虽然强调了 ChromaDB 与 Reranker，但当前计划缺少 embedding 选型，确实不完整。

**计划结论**：

- 开发默认使用：`BAAI/bge-small-zh-v1.5`
- 评测/冲刺阶段可升级：`BAAI/bge-large-zh-v1.5`

这样做的原因：

1. `small` 更利于开发和调试
2. `large` 更利于后期冲精度
3. 二者可共用同一套 `Embedder` 接口

### 3.3 建议：补齐延迟预算

**判断：完全可采纳**

原因：

1. 比赛说明里写明 `语音响应延迟 < 5 秒`
2. 顶层设计文档也明确提出 `Full-Streaming`
3. 如果现在不把检索、重排、首 token 的预算定下来，后面很容易做出“准确率还行但总耗时超标”的方案

**计划结论**：

- 下一阶段必须把延迟预算正式纳入计划与验收

### 3.4 建议：明确多模态模型方案

**判断：可采纳，但要稍作边界修正**

原因：

1. 比赛要求“至少 1 个多模态大模型作为核心 AI 能力支撑”
2. 当前计划只写了“真实模型配置”，没有说明如何满足这一点，确实不够

但需要修正的地方是：

1. “选定多模态模型”可以纳入下一阶段计划
2. “立即把前端图片上传与图像问答一起做掉”不属于下一阶段的最低必要范围

**计划结论**：

- 下一阶段接入的真实 LLM 应优先选择多模态模型
- 推荐：`Qwen-VL-Plus`
- 备选：`DeepSeek-VL2`
- 本轮先保证：
  - 后端模型本身具备多模态能力
  - 文本问答链路先打通
- 图片输入前端与图像问答交互不作为这一轮硬交付

这既符合题目要求，也控制了范围。

### 3.5 建议：评测集固定为 100 题，并明确判定标准

**判断：完全可采纳**

原因：

1. 当前计划只说“做评测闭环”，没有规模和标准，确实过于抽象
2. 如果没有固定题量与判定方式，后续准确率没有基线可比

**计划结论**：

- 采用 `100` 题评测集方案：
  - FAQ `40`
  - 知识 `40`
  - 推荐 `20`
- 判定方式：
  - 主判定：`must_contain`
  - 辅判定：`LLM-as-Judge`

## 四、修正后的阶段目标

下一阶段不再简单表述为“RAG + LLM + Eval”，而是明确为：

**真实 RAG + 真实 Reranker + 多模态 LLM 接入 + 评测闭环**

## 五、修正后的实施计划

### Phase 1：真实 RAG + 真实 Reranker

#### 目标

将当前的占位检索链路升级为：

`切块 -> 向量化 -> Chroma 检索 -> Cross-Encoder 重排 -> 返回证据`

#### 要做的事

1. 新增 `KnowledgeChunk` 数据模型
   - 建议字段：
     - `chunk_id`
     - `source_entry_id`
     - `chunk_index`
     - `content`
     - `metadata_json`

2. 实现切块脚本
   - 面向 `knowledge_entries` 与 `guide_sections`
   - 保留景点名、章节类型、综述类型等 metadata

3. 实现 `Embedder`
   - 开发默认：`bge-small-zh-v1.5`
   - 冲刺可切：`bge-large-zh-v1.5`

4. 接入 `ChromaDB`
   - 替换当前仓储搜索占位
   - 保留 metadata 回传

5. 实现真实 `Reranker`
   - 首选：`bge-reranker-base`
   - 形式：`Cross-Encoder`

6. 保留 `QueryRewriter`
   - 继续作为检索前规范化层

7. 视时间补 `FAQ` 语义匹配 L3
   - 可复用 embedding 服务
   - 不抢占主链路优先级

#### 预期交付

1. 可运行的切块构建脚本
2. 可运行的向量索引构建流程
3. 可运行的真实检索服务
4. 可运行的真实重排服务

#### 验收方式

1. FAQ 未命中后不再走仓储搜索占位
2. 检索 Top-5 中能包含正确证据
3. Reranker 输出顺序会真实受 query 影响

### Phase 2：真实多模态 LLM 接入

#### 目标

让当前问答生成阶段由真实模型接管，并满足“至少 1 个多模态大模型”的要求。

#### 要做的事

1. 明确模型方案
   - 推荐：`Qwen-VL-Plus`
   - 备选：`DeepSeek-VL2`

2. 保持现有 `OpenAI-compatible` 适配层结构不变

3. 设计首版 Prompt
   - 只依据检索证据回答
   - 证据不足时明确说明不足
   - 严禁脱离资料包自由发挥

4. 替换当前占位流式回答实现

5. 保持现有 SSE 流式输出结构不变

#### 预期交付

1. 真实模型配置接入能力
2. 问答接口输出真实模型回答
3. 回答与检索证据保持一致

#### 验收方式

1. FAQ 未命中后能得到真实生成回答
2. 回答内容与检索证据一致
3. 证据不足时不编造
4. 模型配置满足“多模态”要求

### Phase 3：评测闭环

#### 目标

建立准确率与延迟的双评测体系。

#### 要做的事

1. 构建 `100` 题评测集
   - FAQ `40`
   - 知识 `40`
   - 推荐 `20`

2. 统一评测字段
   - `id`
   - `category`
   - `query`
   - `expected_answer`
   - `must_contain`
   - `source_doc`

3. 增加准确率评测脚本
   - 主判定：关键词命中
   - 辅判定：`LLM-as-Judge`

4. 增加延迟评测脚本
   - FAQ 精确匹配
   - 检索
   - 重排
   - 首 token
   - 全链路

5. 增加回归入口脚本

#### 预期交付

1. 首版 100 题评测集
2. 准确率脚本
3. 延迟脚本
4. 回归入口
5. 首轮评测基线

## 六、延迟预算

### FAQ 路径

1. Query 改写 `< 10ms`
2. FAQ 精确匹配 `< 5ms`
3. FAQ 模糊匹配 `< 20ms`
4. FAQ 命中全链路 `< 600ms`

### RAG 路径

1. Query 改写 `< 10ms`
2. Embedding + Chroma 检索 `< 300ms`
3. Reranker `< 150ms`
4. LLM 首 token `< 2000ms`
5. 当前轮总链路 `< 2500ms`

### 后续全链路目标

在未来接入 TTS 与 Avatar 后，最终全链路目标为：

`< 5000ms`

## 七、为什么这样排优先级

1. 准确率是比赛成败的第一核心
2. 当前最弱的是检索、重排、生成三段
3. 语音和数字人如果先做，只会更快、更漂亮地输出占位答案
4. 先把真实问答主链路做实，再接语音和数字人才更稳

## 八、下一阶段明确不做

1. 不接真实 ASR
2. 不接真实 TTS
3. 不接真实 Avatar
4. 不做前端视觉大改
5. 不做复杂数据大屏
6. 不做上传工作流
7. 不把行为分析数据混入问答事实源

## 九、风险点

1. `Cross-Encoder` 与 embedding 模型会引入新依赖和模型下载成本
2. 多模态模型最终供应商需要你确认账号与 API
3. 切块策略不稳会直接影响准确率
4. `bge-large` 可能只适合冲刺评测，不适合开发期常驻
5. 如果没有 100 题评测集，后续所有优化都缺少客观基线

## 十、结论

DeepSeek 的审查建议整体可行，而且与最上层架构设计文档高度一致。真正需要做的不是推翻当前计划，而是把它从“方向正确”修正为“可落地、可验证、可冲指标”。

修正后的下一阶段计划是：

**先做真实 RAG + 真实 Reranker，再接多模态 LLM，随后建立 100 题评测闭环。**

这条路径最符合当前题目要求，也最符合我们现在项目的实际进度。
