# 2026-06-26 设计契约接入与运营报表升级报告

## 1. 本轮目标

在不改动核心业务接口的前提下，先完成两件事：

1. 为项目建立一份可复用的页面设计契约，减少后续页面继续“各做各的”。
2. 将管理后台 `AnalyticsReport` 从基础图表页升级为适合软件杯答辩展示的运营分析页。

## 2. 为什么先做这一轮

结合赛题要求和当前评委视角评分，现阶段最需要强化的是：

- 管理端的“可运营感”和“完成度观感”
- 页面间视觉统一，而不是继续堆新功能入口
- 让评委在短时间内看懂“游客服务数据 + 知识盲区闭环”

因此，本轮不优先上重型 Figma 流程，而是先把设计约束写进仓库，并用一个关键页面验证整套方法。

## 3. 本轮改动

### 3.1 新增项目级设计契约

新增文件：`DESIGN.md`

核心内容包括：

- 项目定位：灵山智慧导游，而非通用 AI 助手
- 游客端与管理端的不同目标和语气
- 色彩、字体、图表和表格的统一约束
- 反 AI 味清单
- Codex 后续改页面时的执行约定

作用：

- 后续不管是继续改 `KnowledgeManage`、`BlindSpotManage`，还是再优化游客端，都有统一判断标准。

### 3.2 升级运营分析页

修改文件：`frontend/src/views/admin/AnalyticsReport.vue`

本次升级点：

1. 增加 Hero 头部，明确这是“运营分析与知识盲区”页面。
2. 增加 4 张摘要卡，先展示咨询量、响应时长、游客规模、待补高频问题。
3. 图表区全部改成带标题、说明和总量标识的答辩型卡片。
4. 将“问答路径分布”从普通饼图改为更易读的横向条形图。
5. 为游客群体分析增加“热门偏好”标签，增强解释性。
6. 将知识盲区表格和运营指标快照并列展示，形成“数据观察 -> 运营动作”闭环。
7. 新增局部失败提示与刷新时间展示，增强后台可信感。

## 4. 验证结果

执行：

```bash
cd frontend
npm run build
```

结果：

- 前端生产构建通过
- 未出现本轮改动引入的语法或打包错误
- 仍存在既有 chunk 体积警告，但不属于本轮新增问题

## 5. 当前收益

本轮完成后，仓库里已经有了一个明确的设计执行基准，且管理后台中至少有一个页面被拉到了更接近答辩展示的质量线。

这意味着下一步继续改：

- `frontend/src/views/admin/KnowledgeManage.vue`
- `frontend/src/views/admin/BlindSpotManage.vue`

时，不需要再重新讨论“风格到底往哪走”，可以直接照 `DESIGN.md` 和升级后的 `AnalyticsReport.vue` 继续推进。

## 6. 下一步建议

1. 继续升级 `KnowledgeManage.vue`，把同步、重建索引和新增知识做成更清晰的运营工具栏。
2. 升级 `BlindSpotManage.vue`，强化待处理/已解决切换、补录入口和高频问题提示。
3. 最后做一次管理端三页联调，确保 `Dashboard / Analytics / BlindSpots / Knowledge` 的视觉语言一致。
