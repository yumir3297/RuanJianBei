# 2026-06-11 官网要求与 GitHub 借鉴后的开发重排计划

## 1. 本次重排原因

用户提供赛题官网：

https://www.cnsoftbei.com/content-3-1245-1.html

对照官网后发现，之前计划对“准确问答主链路”把握正确，但对以下内容显式优先级不足：

1. 管理后台完整度。
2. 数字人可演示 MVP。
3. 语音输入、语音输出、表情、口型同步的产品侧闭环。
4. 游客反馈分析与运营数据大屏。
5. 7 分钟演示视频所需的可见功能闭环。

因此本计划在不推翻当前 RAG 优先策略的前提下，将后台与数字人从“最后有空再做”调整为“真实问答链路之后必须补齐的正式交付线”。

## 2. 官网关键要求重新提炼

### 2.1 游客交互端

必须覆盖：

1. 文本输入。
2. 语音输入。
3. 数字人语音回答。
4. 数字人口型同步。
5. 数字人表情配合。
6. 景区历史、文化、景点特色问答。
7. 个性化路线推荐。
8. 根据游客兴趣生成路线和讲解重点。

### 2.2 管理后台端

必须覆盖：

1. 知识库管理：上传、更新、维护讲解词、文史资料、FAQ。
2. 数字人形象管理：外观、服装、声音等配置。
3. 游客感受度报告：关注点、情感趋势、服务建议。
4. 数据大屏概览：当日/本周服务人次、热门问答、游客满意度趋势。

### 2.3 非功能指标

必须覆盖：

1. 至少 1 个多模态大模型。
2. 本地景区知识库。
3. 事实性问答准确率不低于 90%。
4. 语音问答延迟小于 5 秒。
5. 系统稳定，不崩溃、不长时间无响应。
6. 数字人自然度由演示视频专家评估。

### 2.4 提交物影响

初赛提交要求包括：

1. 源代码或可执行文件。
2. 部署和使用手册。
3. 总体设计文档。
4. 方案介绍 PPT。
5. 不超过 7 分钟演示视频。

用户已明确当前只要求代码创作，因此本计划只安排代码开发，但代码阶段必须服务于后续演示视频可录制。

## 4. GitHub 可借鉴方案

### 4.1 RAG / 知识库 / 管理后台

## 5. 借鉴原则

1. 不整体搬大型项目。
2. 只借鉴架构、模块边界、页面组织和接口设计。
3. 不因为接数字人而破坏现有 RAG 主链路。
4. 重依赖项目必须后置验证，不能成为当前阻塞点。
5. 演示 MVP 优先于复杂真实渲染。
6. 代码必须保持模块性、可添加性、可阅读性。

## 6. 重新安排后的开发顺序

### Phase 1：真实 RAG 索引构建

目标：

1. 下载 `BAAI/bge-small-zh-v1.5` 到 D 盘缓存。
2. 用当前 `KnowledgeChunk` 构建真实 Chroma 索引。
3. 验证索引数量和 metadata。

原因：

1. 这是准确问答的地基。
2. 当前依赖已装好，正好进入这一阶段。

验收：

1. `indexed_chunks == db_chunk_count`。
2. Chroma 数据位于 `D:\桌面\软件杯\kb\chroma_db`。
3. 模型缓存位于 `D:\桌面\软件杯\.cache\huggingface`。
4. 重复构建不会产生脏数据。

### Phase 2：VectorBackedRAGService

目标：

1. 用 Chroma 替换 SQL LIKE 占位检索。
2. FAQ 未命中后进入真实向量检索。
3. 空索引时显式 fallback 到旧 SQL 检索。

验收：

1. 返回 `RetrievedDocument.content` 完整文本。
2. sources 展示 `snippet`。
3. Chroma Top-20 检索目标 `< 300ms`。
4. fallback 有明确 warning，不伪装成真实 RAG。

### Phase 3：30 题检索评测 v0

目标：

1. 在 LLM 前独立评估检索质量。
2. 区分“召回问题”和“生成问题”。

验收：

1. 建立 30 题检索评测集。
2. 输出 Recall@5。
3. 输出 MRR。
4. 输出检索耗时。

### Phase 4：真实 Reranker + BM25 fallback

目标：

1. 接入 `BAAI/bge-reranker-base`。
2. 使用 `query + doc.content` 做真实重排。
3. CrossEncoder 不可用时，显式使用 BM25 fallback。

验收：

1. 不同 query 会改变排序。
2. 输出真实 score。
3. 记录 rerank latency。
4. 目标 `< 150ms`，不达标则记录真实基线。

### Phase 5：真实多模态 LLM 接入

目标：

1. 替换占位回答。
2. 使用至少一个多模态大模型作为核心 AI 能力支撑。
3. 保持 SSE 输出。

候选：

1. `Qwen-VL-Plus`
2. `DeepSeek-VL2`

验收：

1. system prompt 可配置。
2. 回答只基于检索证据。
3. 证据不足不编造。
4. 首 token 目标 `< 2s`。

### Phase 6：管理后台 MVP 补全

目标：

把当前后台从“骨架页面”补成官网要求的可演示 MVP。

内容：

1. 知识库管理：
   - 列表
   - 新增
   - 编辑
   - 删除或禁用
   - 重新同步 processed data
   - 重新构建 RAG 索引
2. 数字人配置：
   - 外观方案
   - 声音方案
   - 表情策略
   - 口型策略
   - 当前 provider 状态
3. 数据大屏：
   - 当日服务次数
   - 本周服务次数
   - 热门问答
   - FAQ/RAG 命中占比
   - 平均延迟
   - 游客满意度或情感趋势占位指标
4. 游客感受度报告：
   - 高频关注点
   - 情感倾向
   - 服务建议

验收：

1. 管理后台可展示真实 chat log 统计。
2. 可以触发数据同步和索引构建。
3. 可以配置数字人展示参数。
4. 有可用于演示视频的后台画面。

### Phase 7：游客端多模态交互 MVP

目标：

把游客端从文本问答升级为官网要求的基础多模态交互。

内容：

1. 语音录制按钮从占位变成真实录音。
2. 增加 ASR endpoint。
3. 文本进入现有 FAQ/RAG/LLM 链路。
4. TTS 生成语音回答。
5. 前端播放回答音频。

验收：

1. 游客可以语音提问。
2. 系统可以语音回答。
3. 文本问答仍可用。
4. 语音问答总延迟向 `< 5s` 目标靠拢。

### Phase 8：数字人 Avatar MVP

目标：

先做可演示、轻量、稳定的数字人 MVP，再考虑接重型开源数字人。

第一版建议：

1. 使用 2D Avatar 展示。
2. 根据 TTS 播放状态做口型开合动画。
3. 根据回答状态做表情变化：
   - idle
   - thinking
   - speaking
   - happy
   - apology
4. 和文本、来源、路线推荐并列展示。

后续增强：

1. 参考 OpenAvatarChat 做 ASR/LLM/TTS/Avatar pipeline。
2. 参考 SadTalker 或 Wav2Lip 做视频级口型增强。

验收：

1. 数字人能随回答播报。
2. 口型和语音大致同步。
3. 表情随状态变化。
4. 即使 Avatar 失败，文本问答仍可用。

### Phase 9：100 题端到端评测 v1

目标：

建立最终准确率与延迟基线。

内容：

1. FAQ 40 题。
2. 知识问答 40 题。
3. 路线推荐 20 题。
4. must_contain 命中率。
5. 推荐路线命中率。
6. 延迟 P50/P95。

验收：

1. 输出可复现报告。
2. 事实性问答目标向 90% 靠拢。
3. 能定位 FAQ、RAG、Reranker、LLM 哪段拖后腿。

### Phase 10：演示闭环打磨

目标：

保证代码功能能支撑 7 分钟演示视频。

内容：

1. 准备典型游客问题。
2. 准备路线推荐演示。
3. 准备语音问答演示。
4. 准备数字人播报演示。
5. 准备后台大屏演示。
6. 准备知识库更新和重建索引演示。

说明：

本阶段仍只做代码和可运行功能，不写 PPT 和讲稿。

## 7. 当前立即要做的下一步

根据当前进度，下一步应继续：

**Phase 1：构建真实 Chroma 索引。**

具体动作：

1. 运行数据同步，确保 `KnowledgeChunk` 最新。

2. 检索评测 v0

## 8. 需要用户确认的点

1. 是否允许下一步首次下载 `BAAI/bge-small-zh-v1.5`。
2. 是否继续坚持所有模型缓存与向量库都放在 D 盘项目目录。
3. 后续多模态 LLM 是优先接 API，还是优先尝试本地开源模型。
4. 数字人 MVP 是否接受先做轻量 2D Avatar，再评估 SadTalker / Wav2Lip / OpenAvatarChat。

## 9. 结论

新的优先级不是“RAG 和数字人二选一”，而是：

1. 先把 RAG/LLM 的回答质量做真。
2. 紧接着补后台 MVP。
3. 再补语音和数字人 MVP。
4. 最后用评测和演示闭环把作品打磨成可提交状态。

这条路线更贴合官网评分项：

1. 功能完整度。
2. 技术与创新性。
3. 行业实用与体验性。
4. 文档和演示可展示度。
