# 阶段 D1：浏览器语音识别接入执行计划

## 1. 当前进度判断

截至 2026-06-19，项目已经完成文本问答、官方资料 RAG 证据链、主动选择、多轮追问、知识盲区管理、Qwen 图片识别与游客端图片入口。语音部分目前只有：

1. 前端 `MediaRecorder` 录音骨架。
2. `POST /api/asr/transcribe` 上传接口。
3. 后端 `StubASRService` 占位转写。

因此当前“语音输入”并不是真实语音识别。本轮进入阶段 D1，用浏览器 Web Speech API 补齐零成本真实中文转写。

## 2. 本轮目标

实现以下闭环：

```text
点击语音输入
-> 浏览器请求麦克风权限
-> 中文语音实时转成文字
-> 文字显示在现有问题输入框
-> 用户确认后按 Enter 或点击按钮发送
```

## 3. 实施内容

### D1.1 独立语音识别模块

新建 `frontend/src/composables/useSpeechRecognition.js`：

1. 检测 `SpeechRecognition` / `webkitSpeechRecognition`。
2. 使用 `zh-CN`、临时结果和单次识别模式。
3. 分别维护实时文本、最终文本、监听状态和错误状态。
4. 支持开始、停止和页面卸载清理。
5. 连续累积最终识别片段，避免后一个片段覆盖前一个片段。

### D1.2 游客端接入

修改 `frontend/src/views/tourist/ChatView.vue`：

1. Chrome 支持时优先使用 Web Speech API。
2. 识别中的临时文字实时同步到问题输入框。
3. 停止后保留最终识别文字，由游客确认后发送。
4. 显示“正在聆听”和可读错误提示。
5. 浏览器不支持时保留现有 MediaRecorder 上传降级。
6. 文本输入始终可用。

### D1.3 审查与验证

1. 审查模块状态清理、重复启动、异常结束和空结果路径。
2. 运行 `npm.cmd run build`。
3. 启动本地前端后检查按钮、状态提示和输入框交互。
4. 麦克风真实识别需要在 Chrome 中由用户授权后人工说话验收。

## 4. 明确不做

1. 不修改后端。
2. 不替换 `StubASRService`。
3. 不新增 npm 依赖。
4. 不修改 SSE 协议、数据库或公共 API。
5. 不在识别结束后自动发送问题。
6. 不把语音识别文本作为事实来源，只作为游客输入。

## 5. 验收标准

1. 支持 Web Speech API 的 Chrome 中显示“语音输入”按钮。
2. 点击后进入聆听状态，语音文字实时进入输入框。
3. 点击结束或浏览器自动结束后，最终文字仍保留。
4. 用户可按 Enter 或点击“发送问题”进入原有问答链路。
5. 拒绝麦克风权限、无语音、网络异常时页面不崩溃并显示提示。
6. 不支持 Web Speech API 时仍可使用原有录音降级或文本输入。
7. 前端生产构建通过。
