# 第 1 章 项目概述

## 1.1 项目背景

灵山胜境的讲解内容横跨景点信息、佛教文化、历史沿革、演出活动、游览路线和服务设施。游客在现场通常只有零散时间获取信息，传统服务方式存在三类明显不足：

1. 人工导游服务容量有限，旺季难以覆盖大量散客。
2. 固定语音导览只能按预设顺序播放，无法回答游客临时产生的问题。
3. 普通大模型虽然能自由对话，但容易混入未经景区核验的信息，回答过程也难以追溯。

景区管理端同样面临问题：游客反复询问了什么、哪些问题知识库答不上来、哪些讲解内容需要更新，往往依赖人工观察，难以形成稳定的数据闭环。

因此，本项目没有把目标停留在“做一个会说话的数字人”，而是尝试建立一套完整的景区知识服务系统：前台负责自然交互，后台负责知识维护，评测体系负责证明回答质量。

## 1.2 产品定位

本系统是一套面向灵山胜境的双端 AI 导览产品：

- 游客端提供文本问答、主动选择、拍照识景、语音输入和数字人展示。
- 管理端提供知识维护、知识盲区处理、数字人配置、服务统计和运营分析。
- 问答内容优先使用景区官方资料，通过检索证据约束模型回答。

一句话概括：

> 游客得到的是随时可问、回答有出处的数字导游；景区得到的是可以持续补充和改进的知识服务后台。

## 1.3 建设目标

### 1.3.1 游客体验目标

- 不要求游客先学会“怎样向 AI 提问”，可以通过景点、话题和路线按钮逐步选择。
- 回答正文展示 `[证据N]`，并在页面右侧提供来源摘要。
- 支持连续追问，系统能够保持当前景点或话题主体。
- 图片识别只负责判断“可能看到了什么”，最终讲解仍回到官方资料。
- 数字人和语音能力发生故障时，文本问答仍可使用。

### 1.3.2 景区运营目标

- 官方资料可以同步、拆分并建立向量索引。
- 无证据问题自动沉淀为知识盲区，而不是由模型随意补全。
- 管理员补录 FAQ 后，问答索引能够随之刷新。
- 管理后台能够查看服务次数、命中路径、平均延迟和高频问题。

### 1.3.3 比赛验收目标

- 至少有一个多模态大模型真实参与业务链路。
- 景区事实问答准确率目标不低于 90%。
- 语音问答端到端延迟目标小于 5 秒。
- 数字人具备可展示的说话状态、口型和表情变化。
- 作品具备可运行程序、设计文档、部署手册、方案 PPT 和演示视频。

## 1.4 本项目重点解决的问题

| 问题 | 常见做法 | 本项目的处理 |
|---|---|---|
| 大模型会编造景区事实 | 直接让模型回答 | 先检索官方资料，再用编号证据约束生成 |
| 游客问题短且含糊 | 只做关键词匹配 | 主动选择上下文、文字实体和历史主体联合消歧 |
| 图片识别结果可能出错 | 直接把视觉模型输出当答案 | 视觉结果只作为检索线索，事实仍由官方资料提供 |
| 知识库答不上来 | 返回通用兜底话术 | 自动聚合知识盲区，支持后台补录并刷新索引 |
| 数字人链路较重、容易拖垮页面 | 所有能力强耦合 | 文本、音频和 Avatar 分层降级，保证核心问答可用 |
| 比赛指标难以证明 | 只展示几个成功案例 | 建立固定测试集、检索评测和端到端评分脚本 |

## 1.5 系统边界

本阶段纳入范围：

- 灵山胜境官方资料的提取、清洗、入库、切片和检索。
- 文本问答、FAQ 快速通道、RAG 检索和证据引用。
- 景点、话题、路线主动选择及连续对话。
- Qwen 视觉模型参与的图片识别与官方资料讲解闭环。
- 浏览器中文语音识别入口。
- Live2D 数字人展示、基础表情和说话状态。
- 知识库、知识盲区、数字人配置和数据概览后台。
- 检索、FAQ 语义匹配和端到端评测工具。

本阶段不纳入或仅作方案预留：

- 景区票务、支付、闸机和会员系统对接。
- GPS、Wi-Fi 指纹和蓝牙 Beacon 的真实硬件部署。
- 原生 Android / iOS 应用。
- 多景区、多租户和大规模分布式部署。
- 已达到生产质量的真实 TTS、音频驱动口型和声纹克隆。

## 2.2 得分点组织

本项目围绕四条主线组织设计与演示：

1. **功能完整度**：游客端和管理端不是两套孤立页面，而是共享同一套知识、日志和配置数据。

# 第 3 章 用户与使用场景

## 3.1 用户角色

### 游客

游客可能是第一次到访，也可能对佛教文化、建筑艺术或亲子路线有明确兴趣。其共同特点是使用时间短、输入不规范、对错误信息容忍度低。

主要操作：

- 直接输入问题。
- 选择游览目标、景点、话题或路线。
- 拍摄眼前景物，请系统识别并讲解。
- 使用语音把问题转成文字。
- 查看回答引用的景区资料。

### 景区运营人员

运营人员不需要理解向量数据库或大模型参数，主要关心内容是否正确、哪些问题答不上来、当天服务情况如何。

主要操作：

- 新增、编辑或删除知识条目。
- 同步官方资料并重建索引。
- 查看和处理知识盲区。
- 切换数字人预设配置。
- 查看服务概览和高频问题。

## 3.2 典型场景

### 场景一：事实问答

游客问：“灵山大佛有多高？”

系统优先命中 FAQ 快速通道，返回结构化答案、`[证据1]` 和官方资料来源。该路径无需等待大模型生成，适合高频、确定性问题。

### 场景二：文化讲解

游客问：“玄奘为什么把马山称为小灵山？”

系统检索官方历史资料，将相关片段交给文本模型组织语言。模型只能基于给定证据回答，并在正文标注证据编号。

### 场景三：指代追问

游客先选择“灵山大佛”，随后问：“它适合带孩子看吗？”

系统通过服务端保存的当前主体理解“它”指向灵山大佛，并把景点范围传给检索器，避免从全库返回其他景点。

### 场景四：拍照识景

游客上传灵山大佛照片。Qwen 视觉模型给出候选景点和画面特征，系统再用这些线索检索官方资料，最终返回可追溯讲解。

### 场景五：知识盲区

游客问：“现在停车场还剩多少车位？”

现有资料没有实时车位数据，系统明确说明无法确定，同时把问题记录为知识盲区。管理员可以在后台决定是否接入实时数据或补充说明。

## 4.3 核心设计原则

### 官方资料优先

景区事实只允许来自官方资料或后台人工核验内容。按钮选择、图片识别结果、历史对话和大模型常识都不能单独成为事实来源。

### 先缩小问题，再调用模型

高频 FAQ 直接回答；明确景点时先限定检索范围；只有需要组织语言和综合多段证据时才调用大模型。这样既降低延迟，也减少幻觉。

### 可降级，不失联

视觉模型、文本模型、语音或数字人失败时，系统按能力层级降级。附加体验可以暂时不可用，但核心文本问答不应随之崩溃。

### 用测试结果约束宣传

准确率、召回率和延迟只使用已保存的测试报告。尚未完成的全量评测在文档中明确标注，不以设计目标代替结果。

### 比赛规模下保持简单

作品采用 FastAPI、Vue、SQLite 和 ChromaDB，优先保证单机可部署、代码易阅读和演示稳定，不提前引入与当前规模不匹配的复杂中间件。

# 第 5 章 核心功能设计

## 5.1 有证据的智能问答

### 5.1.1 分层问答路径

系统没有让所有问题都走同一条昂贵链路，而是采用由快到慢的分层路径：

```text
FAQ 精确匹配
→ FAQ 别名 / 模糊匹配
→ FAQ 语义匹配
→ 选择感知 QA 缓存
→ ChromaDB 向量检索
→ 条件式 CrossEncoder 重排
→ DeepSeek 基于证据生成
```

这样设计有两个直接收益：

- “大佛有多高”一类稳定问题可以快速返回。
- 复杂文化问题仍能由模型组织成自然语言，但内容受证据约束。

### 5.1.2 FAQ 三级快速通道

- L1：标准问题或别名精确匹配。
- L2：受控关键词和模糊匹配。
- L3：使用预计算语义向量，并增加实体、意图双门控。

L3 不直接在全库按相似度“猜答案”。系统先判断问题是否含可信景点实体、是否属于已支持意图，再在候选内排序。当前运行阈值为 `0.60`，由 21 条正例和 13 条盲区负例共同校准。

### 5.1.3 选择感知缓存

缓存键包含：

```text
answer_namespace
+ normalized_query
+ mode
+ attraction_id
+ topic_key
+ route_id
+ knowledge_version
```

同一句“它有什么看点”，在不同景点上下文下不会共用答案。生成器版本或知识版本改变后，旧缓存也不会继续污染新链路。

### 5.1.4 RAG 检索与重排

官方资料先拆分为知识片段，再通过 BGE 模型生成向量写入 ChromaDB。检索时可以按景点和话题 metadata 缩小范围。

CrossEncoder 只在以下情况执行：

- 问题缺少明确景点范围。
- 候选分数接近，存在歧义。
- 问题需要跨景点或跨类别检索。

单候选或明确景点范围内的高置信候选可以安全跳过重排。重排模型不可用时，系统依次降级为 BM25 和原向量顺序。

### 5.1.5 证据约束生成

发送给文本模型的提示包含三条硬约束：

1. 只能使用当前提供的编号资料回答景区事实。
2. 正文必须使用 `[证据N]` 标注引用。
3. 资料不足时明确说明无法确定，不使用模型常识补齐。

前端同时展示来源标题、摘要和原始文件名，评审可以从回答追溯到资料。

## 5.2 连续对话与上下文消歧

### 5.2.1 SelectionContext

```python
class SelectionContext(BaseModel):
    mode: Literal["free_chat", "topic", "attraction", "route"]
    attraction_id: int | None
    topic_key: str | None
    route_id: str | None
    interests: list[str]
    audience_type: str | None
    available_hours: int | None
    avoid_crowded: bool | None
```

### 5.2.2 冲突消解顺序

1. 本轮文字中明确出现、且能在知识库验证的实体。
2. 游客本轮主动点击的景点或话题。
3. 服务端上一轮确认的 active subject。
4. 前端保存的历史选择。
5. 默认全景区范围。

这套顺序解决了一个常见问题：游客已经选择“灵山大佛”，但又在文字中明确询问“梵宫”，系统应以本轮文字为准，而不是继续沿用旧选择。

### 5.2.3 快捷追问

系统按景点、话题、路线和自由提问四类生成 2 至 4 个受控追问。追问来自模板和知识元数据，不把未经核验的新事实写入按钮。

## 5.3 主动选择与路线推荐

游客端提供三类入口：

- 了解景点。
- 规划路线。
- 查询实用信息。

当前数据由后端统一提供，包括 22 个景点、8 个话题和 3 条路线模板。路线推荐结合可用时间、兴趣、人群类型和拥挤偏好进行匹配。

主动选择不是问答页面的装饰，它会进入服务端 `SelectionContext`、缓存键和检索范围，直接影响回答结果。

## 5.4 拍照识景

### 5.4.1 处理流程

```text
游客选择图片
→ 文件类型与大小校验
→ Qwen 视觉模型分析
→ 场景概述 / OCR / 候选景点 / 视觉标签
→ 生成 retrieval_query
→ FAQ / RAG 检索官方资料
→ 返回带证据的景点讲解
```

### 5.4.2 可信边界

视觉模型可能把相似建筑或佛像认错，因此系统明确区分“识别线索”和“景区事实”：

- 候选景点、OCR 和视觉标签可以帮助检索。
- 高度、历史、年代和文化含义必须来自官方资料。
- 识别置信度较低时，前端展示多个候选，由游客确认。

这使多模态模型真实参与业务，又不会绕过知识库直接输出未经核验的讲解。

## 5.5 语音交互

当前游客端优先使用 Chrome Web Speech API：

1. 浏览器获取麦克风权限。
2. 以 `zh-CN` 进行中文识别。
3. 临时识别结果实时显示。
4. 最终文字进入输入框，由游客确认后发送。

保留 `POST /api/asr/transcribe` 作为后端 ASR 扩展接口，但当前 Provider 仍为 Stub。这样既能完成浏览器端语音输入，也为后续接入 FunASR、SenseVoice 或云端 ASR 留出稳定边界。

当前状态说明：

- 浏览器语音识别代码已完成，真实麦克风环境仍需最终人工验收。
- 后端 ASR 不是正式识别服务。
- TTS 当前返回的是占位数据，不是可播放的真实语音。

## 5.6 数字人交互

### 5.6.1 状态设计

| 状态 | 触发时机 | 页面表现 |
|---|---|---|
| idle | 等待游客输入 | 自然待机 |
| listening | 正在接收语音 | 聆听提示 |
| thinking | 正在检索或生成 | 思考状态 |
| speaking | 正在输出回答 | 口型开合 |
| happy | FAQ 等高置信回答 | 微笑表情 |
| apology | 无证据或能力降级 | 歉意表情 |

### 5.6.2 当前实现

- Live2D 模型能够在浏览器加载和缩放。
- PixiJS 与 Live2D 插件版本已完成兼容修复。
- 根据回答路径切换 `happy`、`apology` 等基础表情。
- 模型加载失败时回退到轻量 SVG 数字人。

### 5.6.3 当前限制

口型目前由固定时间间隔控制，不是由真实音频、phoneme 或 viseme 驱动；TTS 仍为 Stub。因此本版本可以展示数字人状态机和降级能力，但“自然语音与口型同步”仍属于最终冲刺项。

## 5.7 管理后台

### 5.7.1 知识库管理

- 查询、新增、编辑和删除知识条目。
- 从处理后的官方资料重新同步数据。
- 重新构建 ChromaDB 向量索引。

### 5.7.2 知识盲区管理

无证据问题按规范化文本去重聚合，保留最多 5 条原始问法样本，并记录出现次数和时间。

管理员可以：

- 查看高频盲区。
- 补录带来源的 FAQ。
- 将盲区标记为已解决。
- 刷新 FAQ 语义索引。

官方资料再次同步时，系统保护已解决盲区关联的人工 FAQ，避免运营成果被覆盖。

### 5.7.3 数字人配置

后台提供三个预设配置，可保存名称、模型路径、预览图和音色类型，并设置当前激活项。当前配置接口已完成，游客端对激活配置的动态读取仍需完善。

### 5.7.4 数据概览

管理总览展示：

- 服务次数。
- 知识条目和 FAQ 数量。
- FAQ / RAG 等命中分布。
- 平均响应延迟。
- 知识资产和近期趋势图表。

其中逐日趋势接口尚未完成，当前页面对演示数据有明确标记，不作为真实运营统计申报。

# 第 6 章 数据设计

## 6.1 数据来源

系统使用赛题提供和团队整理的灵山胜境资料，包括：

- 景点结构化数据集。
- 历史、文化、景点特色与游览指南。
- 景区旅游行为分析数据。

原始 Word、Excel 资料通过脚本提取和清洗，生成结构化 JSON，再同步到 SQLite。知识条目进一步拆分为片段并写入 ChromaDB。

## 6.2 数据关系

```text
knowledge_entries 1 ─── N knowledge_chunks ─── ChromaDB

faq_entries ─── FAQ L1 / L2 / L3 索引

route_templates ─── 路线推荐

chat_logs ─── 服务统计与高频问题

knowledge_blind_spots ─── resolved_faq_id

avatar_configs ─── 当前数字人配置

quick_topics ─── 游客端主动选择
```

## 6.3 核心数据表

| 数据表 | 主要内容 | 设计说明 |
|---|---|---|
| `knowledge_entries` | 标题、分类、正文、来源、别名、metadata | 保存可维护的知识条目 |
| `knowledge_chunks` | chunk id、条目 id、片段正文、来源 | 与 Chroma 向量记录一一对应 |
| `faq_entries` | 分类、别名、答案、来源 | 支撑高频问答和语义快速通道 |
| `route_templates` | 路线名称、时长、行程、讲解点、标签 | 支撑受控路线推荐 |
| `chat_logs` | 原问题、规范化问题、回答、来源、命中路径、延迟 | 用于运营统计和问题追踪 |
| `knowledge_blind_spots` | 规范化问题、样本、次数、状态、解决方式 | 支撑知识补全闭环 |
| `qa_cache_entries` | 规范化键、答案、来源、过期时间 | 降低重复问题延迟 |
| `avatar_configs` | 模型路径、预览图、音色、激活状态 | 管理数字人预设 |
| `quick_topics` | 话题标识、名称、分类、图标、排序 | 后台驱动游客端入口 |

## 6.4 数据一致性

- 知识条目与向量片段通过 `source_entry_id` 关联。
- FAQ 和人工补录内容必须保留非空来源。
- 缓存中同时保存答案和来源，避免只缓存文本导致证据丢失。
- 删除或同步知识后，需要显式重建向量索引。
- 盲区解决记录保留时间和关联 FAQ，便于审计。

# 第 8 章 可靠性、安全与隐私

## 8.1 降级设计

| 故障 | 降级方式 |
|---|---|
| CrossEncoder 不可用 | 使用 BM25，再退回向量原顺序 |
| ChromaDB 不可用 | 保留 SQL / FAQ 等基础路径 |
| 文本模型超时 | 返回明确降级结果，不缓存失败回答 |
| 视觉模型超时 | 保留图片和文字入口，不把失败结果写入事实链路 |
| 浏览器不支持语音识别 | 提示使用录音或文字输入 |
| TTS 不可用 | 继续返回文本和来源 |
| Live2D 加载失败 | 回退到 SVG 数字人 |

## 8.2 输入与接口安全

- Pydantic 对接口请求进行类型和长度校验。
- 图片限制为指定格式，默认最大 5 MB。
- 音频上传最大 10 MB。
- SQLAlchemy 使用参数化查询。
- 管理端写操作由后续部署方案补充登录和权限控制。

## 8.3 密钥管理

- DeepSeek 和 Qwen API Key 仅保存在本地 `.env`。
- `.env.example` 只保留字段名和示例值。
- 测试报告不输出完整 Key。
- 图片真实调用需要显式允许联网，避免评测脚本误消耗额度。

## 8.4 内容可信与隐私

- 图片当前只用于当次识别，不保存游客原图。
- 对话日志用于统计和问题追踪，不要求采集游客真实姓名。
- 无证据时拒答，避免模型凭空生成票价、场次、天气和实时客流。
- 宗教文化讲解以景区资料为准，避免模型随意扩展敏感内容。

# 第 9 章 测试与评测

## 9.1 分层评测思路

只看最终回答，很难判断问题出在检索还是生成。因此项目将评测拆成四层：

1. 检索层：正确知识片段是否进入 Top-K。
2. FAQ 层：常见问题是否命中，盲区是否被错误吸附。
3. 生成层：回答是否包含必要事实、证据编号和来源。
4. 多模态层：图片是否真实经过视觉模型，并回到官方资料链路。

## 9.2 100 题测试集

项目已建立固定 100 题测试集：

| 类别 | 题数 |
|---|---:|
| 事实问答 | 30 |
| 历史文化 | 15 |
| 景点详情 | 15 |
| 路线规划 | 10 |
| 实用信息 | 10 |
| 主动选择 | 8 |
| 连续追问 | 6 |
| 知识盲区 | 6 |

测试集和离线校验脚本已经完成。一次 100 题真实运行因临时后端连续返回 502，结果无效，不能作为准确率结论。最终提交前需要在稳定服务和受控 API 额度下重新执行。

## 9.3 已验证结果

| 评测项目 | 样本 | 结果 | 说明 |
|---|---:|---:|---|
| 原始向量检索 Recall@1 | 30 题 | 96.67% | 未经过重排 |
| 原始向量检索 Recall@5 | 30 题 | 100.00% | 正确片段均进入 Top 5 |
| 原始向量检索 MRR | 30 题 | 98.33% | 首个正确结果排名较靠前 |
| 条件式重排 Recall@1 | 30 题 | 100.00% | `candidate_k=5` |
| 条件式重排 Recall@5 | 30 题 | 100.00% | `max_length=128` |
| 条件式重排 MRR | 30 题 | 100.00% | 推荐配置 |
| FAQ L3 正例准确率 | 21 题 | 100% | 阈值 0.60 |
| FAQ L3 盲区拒绝率 | 13 题 | 100% | 错误命中为 0 |
| 5 题端到端小样本准确率 | 5 题 | 100% | 主要覆盖快路径 |
| 5 题证据引用率 | 5 题 | 100% | 正文均含 `[证据N]` |
| 5 题来源命中率 | 5 题 | 100% | 均返回 sources |
| 图片识别到官方资料问答 | 1 次真实烟测 | 通过 | Qwen 置信度 0.95 |

## 9.4 性能结果

| 环节 | 指标 | 结果 |
|---|---|---:|
| 原始向量检索总耗时 | P50 | 39.50 ms |
| 原始向量检索总耗时 | P95 | 78.15 ms |
| FAQ L3 查询 | P50 | 10.92 ms |
| FAQ L3 查询 | P95 | 13.51 ms |
| 推荐重排配置 | P50 | 661.82 ms |
| 推荐重排配置 | P95 | 833.43 ms |
| 5 题小样本热运行首文本 | P50 | 44.98 ms |
| 5 题小样本热运行首文本 | P95 | 52.74 ms |

说明：

- 5 题小样本多数命中 FAQ、缓存或直答路径，不能代表所有 LLM 问题。
- 图片真实烟测中，视觉识别约 26 秒，完整问答约 32.6 秒，当前适合功能证明，不适合作为实时性能结论。
- 文本模型冷启动和外部 API 网络波动仍是主要延迟风险。

## 9.5 尚未完成的验收

- 100 题真实端到端准确率尚未形成有效报告。
- 真实 TTS 尚未接入。
- 音频驱动口型同步尚未完成。
- “语音输入到数字人播报小于 5 秒”尚未验收。
- 管理端逐日趋势仍缺真实聚合接口。

这些项目将作为提交前的验收清单，而不是在当前文档中提前写成成果。

# 第 10 章 创新点与行业价值

## 10.1 创新点一：视觉模型只负责找线索，知识库负责给事实

很多多模态问答直接把图片交给模型，让模型同时识别和讲解。本项目将两件事拆开：

```text
图片模型：这可能是什么
官方知识库：关于它可以确认什么
文本模型：如何把证据讲清楚
```

这一拆分既满足多模态模型真实参与，也保留了景区事实的可追溯性。

## 10.2 创新点二：主动选择进入检索，而不是停留在界面

游客点击的景点、话题和路线会进入服务端上下文、缓存键和 Chroma metadata 过滤。主动选择因此成为降低歧义和提高速度的一部分，而不只是首页上的推荐按钮。

## 10.3 创新点三：从拒答到知识补全的闭环

系统不把“答不上来”视为一次失败后结束，而是将其变成后台任务：

```text
无证据
→ 规范化并聚合相似问题
→ 运营人员查看频次
→ 补录有来源的 FAQ
→ 刷新索引
→ 后续游客可得到答案
```

这使知识库能随真实游客问题逐步生长。

## 10.4 复杂定位场景预研

灵山梵宫、五印坛城等室内或半室内区域可能出现 GPS 弱信号。项目预留以下分层方案：

1. 室外优先使用 GPS。
2. 室内可使用 Wi-Fi RSSI 指纹匹配。
3. 精确点位可部署 BLE Beacon。
4. 所有情况下保留游客手动选择兜底。

比赛版本只实现手动选择，不申报已完成硬件定位。后续可通过统一 `LocationProvider` 接口接入不同定位来源。

## 10.5 行业价值

对游客：

- 不必下载厚重讲解资料，也不必按固定路线听录音。
- 可以用自然语言、图片或选择按钮获取讲解。
- 能看到回答出处，降低错误信息带来的困扰。

对景区：

- 高频问题由系统分流，减少人工重复讲解。
- 知识盲区和热门问题可被量化。
- 内容更新由后台完成，不需要重新训练整套模型。
- 架构可迁移到博物馆、展馆、古镇和主题公园。

# 结语

本项目最重要的设计选择，不是使用了多少模型，而是给不同模型划定了边界：

- 视觉模型负责观察。
- 检索系统负责找证据。
- 文本模型负责组织表达。
- 数字人负责呈现。
- 管理后台负责让知识持续更新。

这条边界让作品在当前比赛规模下保持轻量，也让每个功能都能被解释、被替换、被测试。最终目标不是让系统“看起来像 AI”，而是让游客敢用、景区能管、评委可以验证。
