# 阶段 A1：受控快捷追问与连续上下文实施报告

- 日期：2026-06-15
- 依据：`设计文档/10-最终总体设计与开发主线.md`
- 状态：已完成并通过验收

## 本轮目标

完成阶段 A1 最后一个切片：回答结束后提供 2–4 个受控快捷追问，并让服务端确认的主体在前端当前会话中持续携带，使“它、这个景点、这条路线”等连续追问保持正确检索范围。

## 实施方法

1. 新增独立的快捷追问生成服务，只读取服务端已经校验的选择模式、主体和话题键。
2. 快捷追问只来自固定模板与已校验元数据，不调用 LLM，不把按钮文字作为事实证据。
3. 在现有 `context` SSE 事件中附加 `conversation_context`，其中 `last_subject` 来自服务端解析结果。
4. 所有 FAQ、缓存、RAG、证据不足路径在 `done` 前统一发送 `followups` SSE 事件。
5. 前端在内存中保存服务端返回的 `conversation_context`，下一轮通过现有 `ChatRequest.context` 原样携带。
6. 前端展示快捷追问按钮，点击后仍走原有聊天请求、选择校验、RAG 与证据链。

## 模块边界

1. `QAPipeline` 只编排事件，不内嵌模板规则。
2. 不新增服务端 Session、Redis 或历史消息表。
3. 不修改数据库 Schema、依赖、模型、Reranker、FAQ L3 或知识盲区逻辑。
4. 不让前端自己推断可信主体；前端只保存服务端返回值。
5. `history_summary` 本轮不自动生成，避免把未经验证的历史文本当作事实或实体依据。

## 预计修改

1. 新增 `backend/app/services/qa/followup_suggestions.py`。
2. 修改 `guided_selector.py`，输出可信 `conversation_context`。
3. 修改 `pipeline.py` 与 `chat.py`，注入生成器并发送 `followups`。
4. 修改前端聊天 Store 与游客聊天页，携带上下文并展示快捷追问。
5. 新增后端模板、事件顺序和 5 轮主体保持测试。

## 验收标准

1. 每次回答后返回 2–4 个快捷追问，且全部来自可审计的固定模板。
2. `context` 事件返回服务端确认的 `last_subject`。
3. 连续 5 轮无显式景点名的追问仍保持同一景点检索范围。
4. 查询中出现新景点时，仍按既有优先级切换主体并更新上下文。
5. 快捷追问点击后携带当前选择与可信上下文。
6. 后端全量测试、编译和前端生产构建通过。

## 实施结果

1. 新增独立 `FollowUpSuggestionService`，按景点、话题、路线和自由提问四类上下文生成 2–4 个固定模板问题。
2. `context` SSE 已增加服务端可信 `conversation_context.last_subject`，不生成未经验证的历史摘要。
3. FAQ、缓存、RAG 和证据不足路径均在 `done` 前发送 `followups` 事件。
4. 前端聊天 Store 会保存可信上下文并在下一轮通过现有 `ChatRequest.context` 携带。
5. 游客端新增“继续探索”快捷追问区，点击后复用原有聊天发送链路。
6. 用户主动选择“清除选择”或“自由提问”时同步清除旧主体与旧快捷追问，避免历史景点重新污染。
7. 后端专项测试 `10 passed`，完整回归 `35 passed`，应用与评测脚本编译通过。
8. 前端生产构建通过，共转换 `1688` 个模块；仅保留已有的大 chunk 警告。
9. 真实 SSE 两轮冒烟通过：第一轮识别 `灵山大佛 / attraction_id=13`，第二轮仅输入“它还有什么特色”仍保持同一主体；两轮均返回 3 个受控追问。
10. 自动化 5 轮连续追问测试全部保持 `source_entry_id=13`，实体保持率为 `5/5`。

## 最终审查

1. 快捷追问来自代码内受控模板和服务端验证后的元数据，没有调用 LLM 自由生成。
2. 连续上下文保存在前端当前页面，不新增服务端 Session、Redis 或数据库字段。
3. 上下文只影响实体解析和检索范围，不作为景区事实证据。
4. 未修改模型、依赖、数据库、Reranker、FAQ L3 和知识盲区逻辑。
5. 当前会话缺少可用的浏览器控制接口，未执行自动化视觉点击检查；已由 Vue 生产构建、真实 SSE 和源码事件检查覆盖功能验收。
