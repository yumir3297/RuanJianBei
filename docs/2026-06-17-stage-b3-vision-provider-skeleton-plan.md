# 2026-06-17 阶段 B3.1 图片识别 Provider 骨架执行计划

## 背景

B2.3 已完成 5 题小样本真实评测闭环，当前主线可以进入多模态能力准备。根据赛题要求，最终作品必须至少有 1 个多模态大模型真实参与业务；根据现有项目约束，图片、ASR 等结果只能作为检索线索，不能作为景区事实来源。

## 本阶段目标

先构建不绑定厂商的图片识别 Provider 骨架，让后续接入阿里云百炼 Qwen 视觉理解或智谱 GLM-5V-Turbo 时有稳定落点。

## 本轮只做

1. 新增 `backend/app/services/vision/` 服务目录。
2. 定义 `BaseVisionService` 抽象接口。
3. 定义 `VisionResult` 结构，明确图片区分为：
   - 场景描述
   - OCR 文本
   - 候选景点名
   - 视觉标签
   - 检索线索
   - 置信度
   - Provider 名称
4. 新增 `StubVisionService`，用于本地开发和自动化测试。
5. 新增单元测试，验证图片识别结果可以转换为 RAG 检索查询，但不会被当作事实来源。
6. 更新当前任务摘要，记录 B3.1 的准备范围。

## 本轮不做

1. 不接入真实阿里云或智谱 API。
2. 不新增 API Key 配置。
3. 不修改公共 HTTP API。
4. 不修改数据库 Schema。
5. 不改前端上传界面。
6. 不把图片识别结果直接写入知识库。
7. 不让图片结果绕过 RAG 证据链直接回答。

## 后续需要确认

1. 真实图片识别 Provider 选阿里云百炼还是智谱。
2. API Key、Base URL、模型名和预算。
3. 前端图片上传入口放在游客聊天区还是单独“拍照识景”入口。
4. 图片识别结果是否要触发自动追问，例如“你可能在看灵山大佛，要我介绍它吗？”。
5. 是否允许保存游客上传图片；默认建议不保存原图，只保存识别摘要和匿名日志。

## 技术原则

1. 图片模型只输出“观察结果”和“检索线索”。
2. 最终讲解仍必须走官方资料 RAG 和证据引用。
3. Stub 可以用于测试，但不能冒充真实多模态能力。
4. 真实 Provider 接入前必须再次生成执行计划并等待确认。
