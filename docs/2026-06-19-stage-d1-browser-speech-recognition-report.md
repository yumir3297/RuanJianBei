# 阶段 D1：浏览器语音识别接入实施报告

## 1. 实施结果

本轮已按 `docs/浏览器语音识别接入-任务书-Codex.md` 完成浏览器 Web Speech API 接入。

当前语音输入顺序为：

```text
Web Speech API（Chrome 首选）
-> MediaRecorder + 现有 ASR 接口（浏览器不支持时降级）
-> 文本输入（始终可用）
```

Web Speech API 识别出的文字只作为游客问题输入，不作为景区事实来源。

## 2. 代码修改

### 新增语音识别模块

新增 `frontend/src/composables/useSpeechRecognition.js`：

1. 支持 `SpeechRecognition` 和 `webkitSpeechRecognition`。
2. 识别语言固定为 `zh-CN`。
3. 支持临时结果和最终结果。
4. 支持分段最终文本累积，避免结果覆盖。
5. 支持开始、停止、异常结束和页面卸载中止。
6. 不新增 npm 依赖。

### 游客端接入

修改 `frontend/src/views/tourist/ChatView.vue`：

1. Chrome 支持时优先使用 Web Speech API。
2. 识别文字实时写入现有问题输入框。
3. 识别期间显示“正在聆听”状态。
4. 识别期间禁止发送，防止临时文本被误提交。
5. 识别结束后保留最终文本，仍由游客确认后发送。
6. 麦克风拒绝、无语音、无设备和网络失败均有中文提示。
7. 浏览器不支持时保留原有 MediaRecorder 降级。
8. 页面卸载时中止未结束的浏览器识别会话。

## 3. 分段审查

### D1.1 composable 审查

1. 重复启动被阻止。
2. 最终结果按结果下标累积。
3. 临时结果在结束后清除。
4. 最终结果在结束后保留。
5. 异常码保留给页面做可读提示。
6. 卸载清理不会继续触发页面回调。

### D1.2 页面集成审查

1. 没有自动发送问题。
2. 没有修改后端、数据库、公共 API 或 SSE。
3. 没有移除文本输入和录音降级。
4. 识别中不能误发问题。
5. 结束后 Avatar 状态恢复为 `idle`。
6. 语音错误不会导致页面崩溃。

## 5. 尚待人工验收

当前桌面会话的内置浏览器测试环境无法创建临时执行资源，因此没有自动点击麦克风权限，也没有代替用户接受权限。

需要在 Chrome 中完成一次人工验收：

1. 打开游客端。
2. 点击“语音输入”。
3. 允许当前站点使用麦克风。
4. 说“灵山大佛有多高”。
5. 确认文字实时进入输入框。
6. 识别结束后按 Enter，确认进入原有问答链路。

在这一步通过前，阶段 D1 状态应记录为“代码完成，真实麦克风验收待完成”。
