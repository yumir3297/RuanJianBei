## 二、审查修正（3 条）

### 修正 1：同步和索引是真实 API，不是 mock

Claude 原方案将同步和索引按钮标注为"演示模式"——**这两个后端 API 真实存在**。

```
POST /api/admin/sync-processed-data   → 从官方资料包同步知识/FAQ/路线到数据库
POST /api/admin/build-rag-index       → 把所有知识向量化写入 Chroma
```

实现方式：在 `api/admin.js` 中新增两个函数直接调用后端，按钮加 loading 状态和结果反馈（用 `ElMessage` 显示成功/失败+耗时）。**不要在按钮文字或提示中标注"演示模式"。**

注意事项：
- `sync-processed-data` 约 2-5 秒，返回导入数量和 FAQ 刷新耗时
- `build-rag-index` 约 10-30 秒，属于重量级操作，需要 `ElMessageBox.confirm` 二次确认
- 两个按钮都放在 KnowledgeManage.vue 的 action-bar 中

```js
// api/admin.js 新增
export async function syncOfficialData() {
  const { data } = await http.post("/admin/sync-processed-data");
  return data;
}

export async function rebuildVectorIndex() {
  const { data } = await http.post("/admin/build-rag-index");
  return data;
}
```

### 修正 2：AvatarConfig 保留路由，改为信息页

不要删除 `/admin/avatar` 路由——App.vue 侧边栏可能已有导航链接。保持路由不变，把 `AvatarConfig.vue` 改为一个 `el-result` 信息页：

```html
<template>
  <section class="panel-card admin-panel">
    <el-result
      icon="info"
      title="数字人配置"
      sub-title="数字人形象、音色、表情策略将在接入真实 Avatar 引擎后激活。当前阶段请关注知识管理和盲区处理模块。"
    >
      <template #extra>
        <el-button type="primary" @click="$router.push('/admin/knowledge')">
          前往知识管理
        </el-button>
      </template>
    </el-result>
  </section>
</template>
```

`el-result` 是 Element Plus 内置组件，用"待激活"的语义替代"功能缺失"，评委不会觉得尴尬。

### 修正 3：faq_id 自动生成的格式

原方案格式：`faq_admin_{timestamp}_{random}`

建议改为更可读的格式：`faq_admin_{盲区 normalized_query 前 20 字}_{timestamp 后 6 位}`

示例：`faq_admin_卫生间在哪里_a3f2b1`

这样将来在数据库中看到 faq_id 就能识别来源，方便运维。

## 三、额外建议（4 条，顺手做）

### 建议 1：时间预算按 3-4 天

Claude 原方案预估 2 天。实际经验中，表单和对话框的边界情况（提交失败后字段是否清空、连续快速点击会不会重复提交、API 超时后 UI 是否卡死、form validation 的实时反馈）会吃掉比预期多的时间。建议按 3-4 天规划。

### 建议 2：Dashboard 增加盲区计数

当前 `GET /api/admin/overview` 没有 `blind_spot_count` 字段。在 Dashboard.vue 的 stat-grid 中增加一个卡片：

```js
// 用盲区列表 API 判断是否有待处理项
async loadBlindSpotCount() {
  const spots = await fetchBlindSpots("open", 1);
  return spots.length > 0 ? spots[0].hit_count : 0;  // 第一项的 hit_count 作为参考
}
```

有盲区时用琥珀色高亮 `el-tag type="warning"`，提醒管理员处理。

### 建议 3：知识管理表单别名用动态列表

Element Plus 没有 `el-tag-input` 组件。用 `el-input` + `el-tag` + `el-button` 组合实现：

```html
<el-form-item label="别名">
  <div class="alias-input">
    <div class="alias-tags">
      <el-tag v-for="(alias, idx) in form.aliases" :key="idx" closable 
              @close="form.aliases.splice(idx, 1)">
        {{ alias }}
      </el-tag>
    </div>
    <el-input v-model="newAlias" placeholder="输入别名后回车添加" 
              @keyup.enter="addAlias" />
  </div>
</el-form-item>
```

### 建议 4：所有 API 调用需要 catch

每个 API 调用的 catch 块中至少显示 `ElMessage.error()`，不要让后端报错时前端白屏：

```js
try {
  await syncOfficialData();
  ElMessage.success("数据同步完成");
} catch (e) {
  ElMessage.error(`同步失败：${e?.response?.data?.detail || e.message}`);
}
```

## 七、后端 API 速查

| 操作 | 方法 | URL | 备注 |
|------|------|-----|------|
| 管理概览 | GET | `/api/admin/overview` | 各种计数 |
| 盲区列表 | GET | `/api/admin/blind-spots?status=open&limit=50` | status=open/resolved |
| 解决盲区 | POST | `/api/admin/blind-spots/{id}/resolve-with-faq` | body 见下方 |
| 同步资料 | POST | `/api/admin/sync-processed-data` | 慢操作，2-5s |
| 重建索引 | POST | `/api/admin/build-rag-index` | 重操作，10-30s |
| 知识列表 | GET | `/api/knowledge/` | - |
| 新增知识 | POST | `/api/knowledge/` | - |
| 更新知识 | PUT | `/api/knowledge/{id}` | - |
| 删除知识 | DELETE | `/api/knowledge/{id}` | - |

**resolve-with-faq 请求体**：
```json
{
  "faq_id": "faq_admin_盲区关键词_xxxxxx",
  "category": "实用信息",
  "aliases": ["游客原始问法1", "游客原始问法2"],
  "answer": "管理员填写的标准答案",
  "sources": ["信息来源"]
}
```
