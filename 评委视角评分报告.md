## 一、赛题核心要求回顾

根据项目设计文档（`10-最终总体设计与开发主线.md`）中归纳的8条"不可退让指标"及官网赛题通用评审框架，综合形成以下评分维度：

| 赛题硬性指标 | 权重推测 |
|-------------|:--------:|
| 1. 至少接入1个真正参与业务的多模态大模型 | 关键 |
| 2. 景区事实性问答准确率 ≥ 90%（固定评测集证明） | 关键 |
| 3. 语音问答端到端响应 < 5 秒 | 关键 |
| 4. 数字人具备语音播报、口型同步和状态/情感表情联动 | 关键 |
| 5. 核心景区事实以官方资料包为第一事实源 | 关键 |
| 6. 游客端支持文本、语音、图片识别和主动选择等交互入口 | 关键 |
| 7. 管理后台支持知识管理、数字人配置、游客分析和数据大屏 | 关键 |
| 8. 全系统具备证据来源、降级通道和可回归验证 | 关键 |

## 二、逐维度评分

### 维度一：创新价值与实用性（满分 35 分）→ 实得 **26 / 35**

| 子项 | 满分 | 得分 | 评审依据 |
|------|:----:|:----:|---------|
| 选题创新性 | 8 | 7 | AI数字人+景区导览是热门赛道，但"灵山胜境"这个落地场景选得好，有真实景区数据和需求。不足之处：AR/VR等更前沿交互未涉及 |
| 技术架构创新 | 8 | 6 | Retrieval-First + Full-Streaming + Graceful Degradation 三层原则体系性强。FAQ L1/L2/L3 + RAG + Reranker 多级检索链路设计科学。但真正的多模态LLM尚未接入（DeepSeek文本模型处于计划阶段），约束了技术上限 |
| 场景实用性 | 10 | 7 | 灵山景区真实数据驱动；主动选择（景点/话题/路线）、冲突纠偏、快捷追问等功能贴近真实游客需求。但语音问答完整链路（真实ASR→TTS）未闭环 |
| 商业/社会价值 | 9 | 6 | 面向真实景区运营可降低人力成本，数据大屏和管理后台有运营价值。但管理后台数据链路多个缺口（盲区0行、游客画像2行、仪表盘硬编码），目前尚无法支撑真正运营 |

**扣分要点**：
- 多模态LLM未接入（-2）：DeepSeek文本API处于代码准备就绪但未执行阶段；图片识别（Qwen-VL）仅在设计文档中提及，未实现
- 语音端到端未闭环（-2）：前端使用浏览器SpeechRecognition做语音输入、浏览器SpeechSynthesis做降级，但真正的后端ASR/TTS API链路未打通
- 管理后台数据不完整（-2）：盲区表0行、游客画像仅2行、仪表盘趋势图硬编码

## 五、冲刺建议（按优先级排序）

### 🔥 第一优先级：修复"一票否决"级问题（2天内）

1. **修复口型同步**：重连 `useAudioPlayer → chat.js → ChatView → Live2DAvatar` 四文件管道（80分钟）
2. **接入真实LLM**：填入DeepSeek API Key，运行B1+B2方案（2小时）
3. **修复3个数字人URL**：让3个形象真正不同（10分钟）

### 📝 第二优先级：补齐管理后台数据闭环（1天）

4. **补齐盲区记录**：在 `chat.py` 无证据路径增加 `KnowledgeBlindSpot` 写入
5. **补齐游客画像**：在聊天管线中每轮调用 `VisitorRepository.upsert()`
6. **修复仪表盘**：`Dashboard.vue` 改为调用 `qa-trend` API

### 📄 第三优先级：竞赛材料准备（2天）

7. **撰写正式文档**：需求规格说明书、概要设计说明书、详细设计说明书
8. **完成100题评测**：扩展评测集并生成评测报告
9. **录制演示视频**：7分钟完整功能演示（需口型修复后）
10. **制作答辩PPT**：突出创新点和技术深度

## 六、一句话总结

> **项目有优秀的架构深度（Retrieval-First + 多级RAG + 评测驱动）和丰富的设计思考，但核心体验缺陷（口型+LLM+数据）使其目前处于"骨架完整、血肉不足"的状态。修复P0级问题后有望冲击80分以上，但留给团队的时间非常紧迫。**
