### 问题 111-2：知识盲区表始终为空（0 行）

- **严重度**：🔴 P0 致命
- **涉及文件**：
  - `backend/app/services/qa/pipeline.py`（第 154-161 行）
  - `backend/app/services/qa/blind_spot_tracker.py`
  - `backend/app/repositories/knowledge_blind_spot.py`

#### 来龙去脉

`knowledge_blind_spots` 表在 349 条聊天日志下始终为 0 行。前端 `BlindSpotManage.vue` 和 `AnalyticsReport.vue` 的盲区 Top10 表格永远显示"暂无数据"。

**根因**：当前 `pipeline.py` 第 154-161 行中，`blind_spot_tracker.record()` 被调用的位置在 FAQ 匹配和缓存命中检查**之后**、RAG 检索**之前**。这意味着：
- 如果问题被 FAQ 命中（快路径），直接 return，永远不会走到盲区记录
- 如果问题命中缓存，也会 return
- 只有当 FAQ 也未命中、缓存也未命中时，才会在 RAG 检索之前调用 record——但此时记录的是"即将进入 RAG 的问题"，而不是"RAG 也无法回答的问题"

**更关键的问题**：`record()` 调用点在代码流中位置不当。它应该记录的是"系统最终无法回答的问题"（即 RAG 返回空 documents 或 LLM 回答置信度不足），而不是"所有进入 RAG 的问题"。

**现状确认**：
- `chat.py` 第 233 行已将 `BlindSpotTracker` 传入 `QAPipeline`
- 管线完整连接无问题
- BlindSpotTracker.record() 本身逻辑正确（去重聚合 + 频次累加）
- 唯一问题是**触发时机**：当前不管 RAG 有没有检索到结果，都会记录。这意味着如果有盲区数据，应该是有的（因为每次非 FAQ/非缓存的提问都会记录）。但表里是 0 行。

需要进一步确认：是否有异常被静默吞掉（`try/except` 已打印日志但未抛出）。检查后端日志是否有 "Knowledge blind spot recording failed" 的输出。

#### 第一版解决方案（仅供参考）

1. 在 `pipeline.py` 中，将盲区记录调用移动到 RAG 检索返回空结果（`if not documents`）的分支内，紧邻 `no_evidence_reply` 的上方。
2. 同时增加第二个触发点：如果 LLM 生成了回答但 finish_reason 为 `fallback_*` 或 `interrupted_*`，也记录盲区。
3. 确认 `KnowledgeBlindSpotRepository.record()` 的 upsert 逻辑不会因为 IntegrityError 而丢失数据（当前有 try/except 重试，应该没问题）。
4. 如果修复后仍无数据，检查 `chat_logs` 中 `hit_level` 字段的分布——确认是否所有问题都被 FAQ 或缓存命中。

**代码审查发现**：这个问题**已经部分修复**。查看当前 `pipeline.py` 代码：

- 第 314-321 行（`_emit_final_answer` 中 FAQ/缓存路径结束后）：也已经有 `visitor_repository.upsert()` 调用

**但是**，两个调用中的参数都是：
```python
self.visitor_repository.upsert(
    session_id=request.session_id,
    interests=[],           # ← 永远为空
    audience_type="general", # ← 永远为 general
)
```

这意味着：
1. 游客画像确实会写入，但每次写入都是 `interests=[]` + `audience_type="general"`，upsert 不会产生新的记录（因为是同一个 session_id）

这不是 bug，是一个设计局限。但对于比赛演示来说，游客画像表有 2 行几乎等于没有数据。

#### 第一版解决方案（仅供参考）

1. **治标方案（推荐）**：编写一个演示数据生成脚本，从 `chat_logs` 表抽取不同 session_id，为每个生成带随机 `audience_type`（亲子/休闲/文化深度/自由）和 `interests`（从 questions 中提取关键词）的 visitor_profile 记录，凑够 30+ 条。
2. **治本方案**：在 `chat.py` 的 `stream_chat` 端点中，从请求的 `persona` 字段提取 audience_type（如果游客在 ModeSelectView 选了"亲子游"，应该用这个信息）。从 query 中提取 interests 关键词。
3. **更简单的演示方案**：在数据库初始化脚本中添加 30 条预设游客画像，确保演示时饼图有内容。

### 问题 333-1：浏览器语音降级时口型同步不精确

- **严重度**：🟠 P1 严重
- **涉及文件**：
  - `frontend/src/composables/useAudioPlayer.js`（playSpeechSynthesis 方法）
  - `frontend/src/stores/chat.js`（audio SSE 事件处理）
  - `frontend/src/components/ThreeAvatar.vue`

#### 来龙去脉

当 TTS 服务不可用或返回 stub 时，系统走浏览器 Web Speech API 降级播放。此时：

1. `useAudioPlayer.js` 的 `playSpeechSynthesis()` 使用 `onboundary` 事件报告播放进度（按字符位置），但**没有生成 `visemeTimeline`**
2. 而后端 `audio` SSE 事件携带的 `viseme_timeline` 是基于真实 TTS 音频时长生成的
3. 降级时，`chat.js` 仍然把后端返回的 `viseme_timeline` 传给 `onSpeechProgress`
4. 但 `viseme_timeline` 的时间轴是基于 TTS 音频时长计算的，与浏览器语音的播放时长不匹配
5. `ThreeAvatar.vue` 在没有有效 `viseme_timeline` 时，fallback 到 `mouthEnergy`（音频能量包络）——但 Web Speech API 不经过 AudioContext，所以 `mouthEnergy` 始终为 0

**最终结果**：降级时数字人嘴巴不动或随机张合。

#### 第一版解决方案（仅供参考）

1. **在 `useAudioPlayer.js` 的 `playSpeechSynthesis()` 中生成伪 visemeTimeline**：
   - 已知：文本 `text` 和预估时长 `estimatedDuration`
   - 简单方案：按字符数均匀分配时间点，每个时间点的 viseme 参数使用拼音首字母判断（"AEIOU"开头为开口，否则为闭口）
   - 伪 visemeTimeline 通过 `onProgress` 回调传递给 chat.js
2. **在 `chat.js` 的 audio 事件处理中**：当 `provider === "stub"` 或 `provider === "bailian_unavailable"` 时，忽略后端返回的 viseme_timeline，改为使用前端生成的伪时间轴。
3. **备选方案**：如果浏览器语音降级，直接将 viseme_timeline 缩减为简单的"播放期间嘴巴持续小幅度开合"效果（不追求音素同步，但至少看起来在说话）。

#### 第一版解决方案（仅供参考）

**最小改动方案**（不引入新模型，不改后端）：

   - `neutral` → 对应设计中的 idle/listening/thinking 三个状态（通过 blink rate 和 head tilt 微调）
   - `happy` → 保持现有微笑
   - `apology` → 保持现有致歉表情
   - 新增 `speaking` 状态（后端已在返回）— 结合口型开合 + 轻微眉毛动作
2. 如果时间允许，在 `useAvatar.js` 中增加 `guide` 状态（配合 wave 手势时使用），映射到 VRM blendShapes

### 问题 333-4：数字人 idle 状态缺乏微动作
